## 5. 核心结算流水线矩阵

## 5.1 主顺序（每回合）

1. 动作写入 `SOURCE`。
2. 调用 `SOURCE_1`（正向 SOURCE -> CUP）。
3. 调用 `SOURCE_マイナス`（负向 SOURCE -> CUP）。
4. 调用 `SOURCE_EXTRA`（素质/状态乘区修正，含部分 DOWNBASE 修正）。
5. 调用绝顶、射精相关后处理（可能继续写 SOURCE/CUP/DOWNBASE/EXP）。
6. 调用 `SOURCE_DOWNBASE`：扣 BASE:体力/気力，并写疲劳。
7. 最后 `PALAM += CUP - CDOWN`，清零 CUP/CDOWN。

## 5.2 关键函数

| 函数 | 作用 | 核心公式 |
|---|---|---|
| SOURCE_REVISION1(x) | 快感系 SOURCE 受对应感覚 ABL 修正 | 分段：0~5 => 10+7x；6~9 => 30+4x；10~15 => 45+2x；16~19 => 70+x；20+ => 90（再除以10使用） |
| SOURCE_REVISION2(x) | 快感触发的欲望增量修正 | 分段：0~5 => 25+7x；6~9 => 45+3x；10~15 => 60+2x；16~19 => 80+x；20+ => 100（再除以100使用） |
| SOURCE_DOWNBASE | 将 DOWNBASE 应用于 BASE | `BASE = LIMIT(BASE - DOWNBASE, 0, MAXBASE)` |

## 6. 体力/气力消耗与恢复矩阵

### 6.1 常规恢复

- `RECOVER(ARG, value, base_name)`：直接加减 BASE，且 clamp 到 `[0, MAXBASE]`。
- `RECOVER_PERMIL(ARG, permil, base_name, use_recover_speed)`：
  - `recover = MAXBASE * permil / 1000`
  - 若启用恢复速度影响：`recover = recover * (TALENT:回復速度 + 4) / 4`

### 6.2 行动后体力/气力扣减

- `DOWNBASE` 在 SOURCE 链路中累计，最终由 `SOURCE_DOWNBASE` 一次性扣除。
- 恋人且特殊状态可降低扣减：
  - 体力乘 `0.7`
  - 气力乘 `0.8`

### 6.3 射精后消耗

- `NOWEX:射精量` 由射精槽、感度等推导。
- `BASE:精力 -= NOWEX:射精量`。
- `DOWNBASE` 追加：
  - 常规：
    - 体力 `+ LIMIT(射精量/8, 10, 100)`
    - 气力 `+ LIMIT(射精量/4, 20, 200)`
  - 剧毒状态：
    - 体力 `+ LIMIT(射精量/4, 30, 300)`
    - 气力 `+ LIMIT(射精量/2, 40, 400)`

### 6.4 绝顶后消耗

- C/V/A/B/M 绝顶分别直接增加 `DOWNBASE`。
- 多重绝顶会先倍率放大，再给额外返还（负 DOWNBASE）:
  - MultipleEc=3：体力-20, 气力-10
  - MultipleEc=4：体力-40, 气力-30
  - MultipleEc=5：体力-80, 气力-70

## 7. 疲劳与睡眠矩阵

### 7.1 疲劳计算

- `CULC_TIRED(x)`:
  - 若 `x < 10`，返回 0
  - 否则 `t = x/10`，返回 `t * sqrt(t)`
- 行动后累加：
  - `ADD_TIRED(ARG, 12*CULC_TIRED(DOWNBASE:体力) + 8*CULC_TIRED(DOWNBASE:気力))`

### 7.2 睡眠影响

- 睡眠中：
  - 睡眠深度随时间、睡眠药强度、酒气、疲劳增长。
  - 疲劳按比例衰减。
- 非睡眠：
  - 睡眠深度按时间下降。

## 8. SOURCE->CUP 关键换算矩阵（首批迁移）

### 8.1 快Ｃ/快Ｂ

- 先做感度修正，再按 `SOURCE_REVISION1` 修正。
- 再由 PALAM:欲情分段乘区，写入 `CUP:快C/B`。
- 同时按 `SOURCE_REVISION2(ABL:欲望)`写入 `CUP:欲情`。

### 8.2 快Ｖ/快Ａ

- 在快Ｃ/快Ｂ基础上，额外吃：
  - 经验分段（EXP:V/A经验 对 EXPLV）
  - 润滑分段（PALAM:潤滑 对 PALAMLV）
- 最终分别写入 `CUP:快V/A` + `CUP:欲情`。

### 8.3 快Ｍ

- 经验分段权重更陡（最高可达 5.0）。
- 同样写 `CUP:快M` + `CUP:欲情`。

## 9. 升级规则矩阵

### 9.1 需求值核心

- `CALC_ABL_DEMAND(abl_id, demand)`:
  - 若倍率未设：用默认倍率
  - 返回：`MAX(1, demand * DEFAULT_ABL_UP_RAIT / ABL_UP_RAIT[abl_id])`

### 9.2 悟性修正

- 当悟性系统开启：`TALENT278_ADJUST = TALENT:悟性`
- 需求等级索引使用：`EXPLV[当前等级 + 偏移 - TALENT278_ADJUST]`
  - 悟性高（正值）=> 需求降低 => 升级更快
  - 悟性低（负值）=> 需求提高 => 升级更慢

### 9.3 技能升级通式

- 多数能力采用：
  - `经验值 > NEED_EXP` 时升级
- 性技类常见：
  - `综合经验 * (10 + 技巧/2 + 天赋修正 + 習得速度) / 10 > NEED_EXP`

## 13. 自然变化矩阵（时间推进）

### 13.1 通用

- 时间步长：`Lapse = TIME_PROGRESS(时间进度管理)`。
- 不同地点、天气、状态会改变：情绪、理性、愤怒、酒气、睡眠深度、疲劳、勃起、药效倒计时。

### 13.2 非互动时被动恢复

- 非诶嘿嘿、非工作、满足条件时：
  - `体力回復量 = (10 + 回復速度) * Lapse / 20`
  - `気力回復量 = (10 + 回復速度) * Lapse / 20`
  - 调用 `RECOVER`

### 13.3 衰弱状态恢复

- `体力回復量 = (3 + 回復速度) * MAXBASE:体力 / 2000`
- `気力回復量 = (3 + 回復速度) * MAXBASE:気力 / 2000`
- 陪睡时双倍。

## 14. 迁移落地矩阵（erAL）

| eraTW 概念 | erAL 建议数据结构 | 迁移要求 |
|---|---|---|
| BASE/PALAM/SOURCE 命名轴 | `StatBlock` 命名键 | 保留键名与数值单位，结算顺序一致 |
| ABL/TALENT/MARK/CFLAG 索引轴 | `IndexedStatBlock` | 必须保持原 `era_index`，避免事件/规则失配 |
| EXPLV/PALAMLV 阈值 | 数据表（TOML） | 支持双曲线切换 |
| SOURCE 链路 | `SettlementService` 分阶段 pipeline | 禁止命令直改 BASE/PALAM，必须先写 SOURCE |
| 疲劳/睡眠 | 独立疲劳模块 | 保留 `CULC_TIRED` 非线性函数 |
| 升级需求 | ABL 升级服务 | 保留 `CALC_ABL_DEMAND` 与悟性偏移 |
| 关系/刻印 | 关系结算服务 | 与 SOURCE 共享输入，不另起平行事件改值 |

## 15. 迁移时必须保持的数值不变量

1. 所有行为先产出 SOURCE，再统一结算。
2. `PALAM += CUP - CDOWN` 必须在后段一次性提交。
3. 体力/气力消耗必须走 `DOWNBASE -> SOURCE_DOWNBASE`。
4. 疲劳必须由体力/气力实际消耗反推，不能固定常数。
5. ABL 升级阈值必须使用 EXPLV 与倍率函数，不可硬编码等级表。
6. 好感/信赖/刻印应与同回合 SOURCE 绑定，避免“脱钩增长”。

## 16. 指令效果与消耗矩阵

### 16.1 指令轴（Train.csv）

- 0~199：调教/亲密/道具/体位/SM 指令
- 300~399：日常互动/关系推进/约会衔接

### 16.2 单指令数值写法（统一范式）

1. 指令函数直接写 `SOURCE:*`（本回合收益/副作用）。
2. 同时写 `DOWNBASE:*`（体力/气力消耗）。
3. 结算阶段统一转换：`SOURCE -> CUP/PALAM`，`DOWNBASE -> BASE`。

### 16.3 消耗强度分层（按显式常量 DOWNBASE 统计）

#### 轻度消耗

- 常见范围：体力 `0~10`，气力 `20~80`
- 典型：爱抚、轻口交、胸部爱抚、低强度道具

#### 中度消耗

- 常见范围：体力 `30~60`，气力 `80~150`
- 典型：常规插入、部分 A 系动作、高频组合技

#### 重度消耗

- 常见范围：体力 `70~120+`，气力 `150~250+`
- 典型：二穴、扩张、针/重度 SM、高压组合技

### 16.4 指令类别到 SOURCE 产出映射

| 指令类别 | 常见 SOURCE 产出 | 常见副作用 |
|---|---|---|
| 爱抚/亲吻/交流 | 快C/B/M、情愛、達成、恭順 | 不潔、逸脱、反感（低~中） |
| V 插入系 | 快V、情愛、性行動、屈従、露出 | 苦痛、反感、DOWNBASE 上升 |
| A 插入系 | 快A、性行動、屈従 | 苦痛/反感更高，DOWNBASE 更高 |
| 奉仕系（手/口/胸/足） | 快C、達成、屈従 | 不潔、反感、气力消耗偏高 |
| 道具系 | 快感或苦痛定向放大 | 反感/恐怖/鬱屈可显著上升 |
| SM 系 | 苦痛、恐怖、征服、加虐 | 反感快速累积，刻印触发率高 |
| 组合/连携（SCOM） | 多通道快感 + 关系项叠加 | 双边或多边 DOWNBASE 提升 |

### 16.5 已生成的全量指令明细

- COMF 全量数值语句索引：`erAL/docs/reference/tw-command-numeric-catalog.md`
- COMF 显式体力/气力消耗区间：`erAL/docs/reference/tw-command-cost-summary.md`
- SCOMF 全量数值语句索引：`erAL/docs/reference/tw-scom-numeric-catalog.md`

说明：以上三表用于“逐指令迁移”，可直接按指令 ID 对照实现。

## 17. TALENT 效果矩阵

### 17.1 TALENT 在数值系统中的三种角色

1. 乘区修正：直接放大/缩小 SOURCE、CUP、DOWNBASE 或恢复量。
2. 阈值修正：改变升级需求、刻印阈值、生理判定阈值。
3. 开关门控：允许/禁用特定结算分支或行为。

### 17.2 高优先级 TALENT（必须先迁移）

| TALENT | 数值作用 |
|---|---|
| 回復速度 | `RECOVER_PERMIL` 与自然恢复按 `(回復速度+4)/4` 线性修正 |
| 悟性 | 升级需求索引偏移：`EXPLV[level + offset - 悟性]` |
| 痛覚 | 苦痛相关消耗乘区：`(10 + 2*痛覚)/10` |
| 濡れやすさ | 润滑相关分段生效速度，间接影响痛苦/反感 |
| C/V/A/B/M感度 | 对应快感 SOURCE 前置乘区 |
| 体型 | V/A 等快感与欲情分流修正 |
| 自制心 | 欲情、屈服、关系增长抑制 |
| 快感応答 | 快感收益与关系增长加权（正/负向） |
| 抵抗 | 好感/信赖/屈从成长抑制 |
| 感情缺乏 | 多类关系与刻印派生收益抑制 |
| 献身的 | 奉仕系收益与关系增长加权 |
| 恋慕/思慕/恋人 | SOURCE_EXTRA 与关系增长乘区核心 |
| 愛欲/淫乱 | 情欲、屈服、反感相关乘区重写 |
| 酒耐性 | 酒气衰减与相关状态波动修正 |

### 17.3 SOURCE 结算阶段的 TALENT 乘区（关键）

- `SOURCE_EXTRA` 中存在大量 TALENT 乘法：
  - 恭順/欲情/屈服/恥情/好意/反感/不快/苦痛/抑鬱 逐项乘区
  - 同一 TALENT 可同时提升正收益并压低负收益（或相反）
- 这类乘区属于“后修正层”，应放在 `SOURCE_1/マイナス` 之后执行。

### 17.4 恢复与体力层的 TALENT

- 回復速度：影响日常恢复、按比例恢复、衰弱恢复。
- 酒耐性：影响酒气变化和相关衍生值。
- 精力超群等生理 TALENT：影响射精节奏与可持续性（通过射精相关函数链）。

### 17.5 成长层 TALENT

- 悟性：统一作用于 ABL/技能升级门槛。
- 習得速度：进入多类性技升级公式，表现为经验等价放大。
- 器用な指/舌使い 等技能天赋：直接提高对应性技成长效率与指令产出。

### 17.6 关系层 TALENT

- FAVOR/TRUST 两条链均受 TALENT 大量修正：
  - 性格组：態度、自尊心、冷漠、開朗/陰鬱
  - 欲求组：快感応答、自制心、抵抗
  - 关系组：思慕、恋慕、恋人
  - 特殊组：魅力/魅惑/謎之魅力（主角侧）

### 17.7 TALENT 迁移实现建议

1. 先实现 `talent_multipliers.toml`：定义每个 TALENT 对各指标的乘区或偏移。
2. 再实现 `talent_gates.toml`：定义指令可用性与结算分支开关。
3. 最后实现 `talent_growth_hooks.toml`：定义升级门槛、经验转换、关系增长修正。

### 17.8 TALENT 细项补全方式

- `Talent.csv` 是轴定义；具体“数值效果”以 ERB 调用点为准。
- 建议迁移顺序：
  1. 先迁移本节高优先 TALENT（会直接影响主战斗/主养成曲线）。
  2. 再迁移门控型 TALENT（只影响部分指令或剧情条件）。
  3. 最后迁移纯风味 TALENT（几乎不改核心数值）。

## 18. 社交与日常指令速查

本节用于快速回答：执行一次指令，到底改了哪些数值。

### 18.1 会话（指令 300）一次会带来什么

#### A. 成功判定

- 结果态：大成功/成功/失败，由随机值与成功率决定。
- 成功率关键因子：
  - ABL:話術技能
  - ABL:教養
  - 一些 DLC 成功率修正

#### B. 直接消耗

- 常规：
  - 目标 `DOWNBASE:気力 += 20`
  - 主角 `DOWNBASE:気力 += 20`
- 若触发“尴尬失败”分支，会提高到 50 或 100。

#### C. 产出的 SOURCE

- 固定起始：`SOURCE:歓楽 = 200`
- 再按 ABL:親密 分段叠加（低亲密低收益，高亲密高收益）。
- 额外写入：
  - `SOURCE:受動 = 100 + 100 * ABL:従順`
  - `SOURCE:征服 = 100 + 100 * ABL:施虐属性`

#### D. 成功/失败倍率

- 失败：歓楽 0.10，征服 0.50，受動 0.50
- 成功：三项 1.00
- 大成功：三项 2.00

#### E. 话术等级倍率（第二层）

- ABL:話術技能 再次乘区：0.20 到 1.50 分段。
- 这意味着同样的会话，话术技能高时好感相关增量会被显著放大。

#### F. 为什么“会话一次加好感”会波动

会话不是直接写 CFLAG:好感度，而是先写 SOURCE。随后在好感结算链中（FAVOR_CALC）被统一转成好感变化。

影响好感变化的主因：

1. 这次会话写入的 SOURCE（歓楽、受動、征服）。
2. 成功态（失败/成功/大成功）带来的倍率。
3. 目标侧 TALENT 与 ABL（如快感应答、自制心、抵抗、亲密、从顺等）。
4. 主角侧魅力类 TALENT 与状态。

### 18.2 泡茶（指令 301）一次会带来什么

#### A. 直接消耗与恢复

- 主角固定消耗：
  - `DOWNBASE:MASTER:気力 += 100`
  - `DOWNBASE:MASTER:体力 += 50`
- 每个在场目标：`DOWNBASE:気力 += 10`
- 法力相关：主角与目标有法力恢复调用（受系统开关影响）。

#### B. 信赖变化

- 指令内先写：`TFLAG:信頼度管理 += 3`
- 其前置成功态还会再给：+2/0/-2
- 所以泡茶一次对信赖通常是正收益，失败时也可能被拉低。

#### C. 产出的 SOURCE

- 固定起始：`SOURCE:歓楽 = 500`
- 従順分段加成：従順越高，歓楽越高。
- 好感度分段加成：CFLAG:2 越高，歓楽越高。
- 额外写入：
  - `SOURCE:受動 = 200 + 100 * ABL:従順`
  - `SOURCE:征服 = 200 + 100 * ABL:施虐属性`

#### D. 成功/失败倍率

- 失败：歓楽 0.10，征服 0.50，受動 0.50
- 成功：三项 1.00
- 大成功：三项 2.00

#### E. 红茶分支

- 消耗道具红茶后，三项 SOURCE 还会再乘 1.5 或 2.0（部分角色可到 2.0）。
- 这是泡茶数值爆发的主要来源。

### 18.3 身体接触（302）与听牢骚（333）怎么查

- 身体接触（302）：
  - 固定写 `SOURCE:歓楽/情愛/露出/反感`
  - 再按亲密与好感分段叠加
  - 成功态决定 0.5/1.0/2.0 级倍率
- 听牢骚（333）：
  - 核心看 `TFLAG:193` 分支（1~6）
  - 不同分支改 `SOURCE:歓楽/情愛/反感` 与心情、愤怒

### 18.4 速查入口（建议配合使用）

- 单指令数值语句全集：`erAL/docs/reference/tw-command-numeric-catalog.md`
- 单指令体力/气力显式消耗区间：`erAL/docs/reference/tw-command-cost-summary.md`
- 组合指令数值语句全集：`erAL/docs/reference/tw-scom-numeric-catalog.md`

使用方法：

1. 先在本节看“该指令属于哪种结算模式”。
2. 再到指令索引里按指令 ID 定位具体语句。
3. 最后回到第 10 节（FAVOR/TRUST）确认怎样转成好感与信赖。

## 19. 大量指令的高效迁移方法

你现在面临的核心问题不是“单条公式不会抄”，而是“指令太多，容易漏、容易不一致”。

下面这套流程是为批量迁移设计的。

### 19.1 先分组，不要逐条硬迁

建议先把指令按数值行为分为 6 组：

1. 社交组：会话、泡茶、身体接触、听牢骚、照料。
2. 奉仕组：手/口/胸/足等。
3. 插入组：V/A 体位及派生。
4. 道具组：振动棒、跳蛋等。
5. SM 组：打、绑、针、恐怖系。
6. 组合组：SCOM 连携。

同组共享一套“基底公式 + 成功倍率 + TALENT 乘区”。先做组模板，再填指令差异，效率会高很多。

### 19.2 用配置表承载指令，不要把参数写死在代码

每条指令至少拆成这些字段：

- command_id / command_name
- preconditions（可执行条件）
- source_add（固定 SOURCE 注入）
- source_mul（成功态倍率、技能倍率、TALENT 乘区）
- downbase_add（体力/气力基础消耗）
- downbase_mul（状态修正，如天气、连发、恋人减免）
- trust_delta / favor_delta（直接项）
- time_cost
- success_tiers（失败/成功/大成功）
- talent_hooks

已经给你放好模板文件：

- `erAL/docs/reference/tw-command-migration-template.csv`

### 19.3 迁移顺序（实战最稳）

1. 先迁移社交组 5 条（300/301/302/333/335），建立完整链路。
2. 再迁移插入组一小批（如 60/61/62/63），验证痛苦修正与 DOWNBASE。
3. 再迁移奉仕组与道具组。
4. 最后迁移 SCOM 组合组。

原因：社交组最容易观察“好感/信赖是否正确”，是最好的回归基线。

### 19.4 每迁移一批都做三类校验

#### A. 单步校验

- 执行 1 次指令，记录：
  - SOURCE 写入
  - DOWNBASE 写入
  - PALAM/BASE 结算后变化

#### B. 连续校验

- 连续执行 N 次（例如 10 次会话），看：
  - 失败率是否逐步上升（会话累计值机制）
  - 好感/信赖变化是否落在预期区间

#### C. 角色差异校验

- 选 3 个 TALENT 组合差异明显的角色，比较同指令输出。
- 若乘区方向相反（应增反减），优先检查 TALENT hooks 顺序。

### 19.5 你现在就能直接用的资料组合

1. 总体规则：`erAL/docs/reference/tw-numeric-layer-matrix.md`
2. COMF 逐指令效果：`erAL/docs/reference/tw-command-numeric-catalog.md`
3. COMF 消耗区间：`erAL/docs/reference/tw-command-cost-summary.md`
4. SCOM 逐指令效果：`erAL/docs/reference/tw-scom-numeric-catalog.md`
5. 迁移填表模板：`erAL/docs/reference/tw-command-migration-template.csv`

### 19.6 最省事的落地方式

如果你希望“最快见效”：

1. 先把社交组填进 CSV 模板。
2. 在 erAL 里做一个 CommandFormulaLoader，按模板驱动结算。
3. 先跑通 300/301/302，再批量导入其余指令。

这样后续新增或调参只改表，不改逻辑代码。
