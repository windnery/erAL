## 3. 权威来源映射

### 3.1 CSV 层

这些文件是“定义通用系统名词表”的直接来源：

- `CSV/Base.csv`：基础状态轴
- `CSV/Palam.csv`：累积成长/情绪轴
- `CSV/source.csv`：行动中间产出轴
- `CSV/Abl.csv`：能力轴
- `CSV/Talent.csv`：素质/性格/特性轴
- `CSV/Mark.csv`：刻印轴
- `CSV/FLAG.csv`：全局持久状态
- `CSV/CFLAG.csv`：角色持久状态
- `CSV/TFLAG.csv`：流程临时状态
- `CSV/Train.csv`：指令编号与显示名

### 3.2 ERB 层

这些文件是“定义系统怎么运行”的主干入口：

- `ERB/COMMON.ERB`
  - 含 `TRAINNAME` / `SELECTCOM` 识别入口
  - 是命令编号和通用函数的重要公共层
- `ERB/MOVEMENTS/MOVEMENT.ERB`
  - 角色移动主循环入口
- `ERB/MOVEMENTS/MOVEMENT2.ERB`
  - 同行、约会、驻留、住宿等位置状态的补充逻辑
- `ERB/イベント関連/BEFORETRAIN.ERB`
  - 指令执行前判定入口
- `ERB/イベント関連/EVENTCOMEND.ERB`
  - 指令结束后事件入口
- `ERB/イベント関連/EVENTCOMEND2.ERB`
  - 指令结束后的扩展事件入口
- `ERB/イベント関連/DATE_CMN.ERB`
  - 约会通用逻辑入口
- `ERB/イベント関連/デート終了タイムアップ処理.ERB`
  - 约会结束/超时处理入口

## 4. 迁移原则

### 4.1 可以直接照搬的部分

- 指令编号与分类
- 通用数值轴的命名和语义
- `SOURCE -> PALAM / BASE / FLAG` 这一类统一结算思路
- 角色状态字段的职责划分
- 日常互动、约会、地点、同行、约会中断等成熟玩法语义

### 4.2 不直接照搬的部分

- 东方题材角色与剧情文本
- 口上内容本身
- 裸数组到处直读直写的实现习惯
- 大量隐式跨文件脚本耦合

### 4.3 口上策略

`erAL` 前期不按 `eraTW` 的做法一开始就给大量角色补专属口上，而是：

1. 先搭建通用口上层。
2. 让所有角色先吃同一套通用文本模板。
3. 等主循环、指令、状态、约会、地点等系统稳定后，再给少量核心角色补专属口上。

这意味着：

- 先迁“口上系统接口和条件链”。
- 后迁“角色专属口上内容”。

## 9. ERB 调用链入口

### 9.1 指令选择

从 `COMMON.ERB` 可以确认：

- `TRAINNAME` 是指令名和编号的重要公共入口。
- `SELECTCOM` 是当前指令选择状态的核心变量。

这说明 `erAL` 后续如果要更像 `eraTW`，应把“指令选择前判定”“指令执行中分支”“指令执行后事件”明确拆开，而不是只保留一个平面命令列表。

### 9.2 移动/驻留/约会位置逻辑

从 `MOVEMENT.ERB` / `MOVEMENT2.ERB` 可以确认：

- 有独立 `CHARA_MOVEMENT()` 主循环。
- 移动系统会检查约会状态。
- 约会中的角色和普通驻留角色并不是同一套刷新规则。
- 同行/约会/招待/住宿等位置状态是移动系统的核心变量，不是 UI 附属物。

### 9.3 指令前后事件

从文件命名和入口可确认：

- `BEFORETRAIN.ERB` 负责指令前判定。
- `EVENTCOMEND.ERB` / `EVENTCOMEND2.ERB` 负责指令后事件。
- `DATE_CMN.ERB` 与 `デート終了タイムアップ処理.ERB` 负责约会流程。

这意味着后续 `erAL` 要继续像 `eraTW` 靠拢时，应优先补齐：

1. `before_command` 前提层
2. `command_result` 结算层
3. `after_command` 事件层
4. `date_common` 约会流程层
