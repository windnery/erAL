# eraTW 口上迁移方案

## 1. 目的

这份文档回答的是：

- `eraTW` 的口上系统是怎么组织的
- 为什么 `erAL` 不应该一开始就复制那种“海量专属口上”体量
- 如何先做通用口上，再补少量角色专属口上

## 2. 本地参考结构

从本地目录能确认这些主干入口：

- `ERB/口上・メッセージ関連/COMMON_KOJO.ERB`
- `ERB/口上・メッセージ関連/KOJO_MESSAGE.ERB`
- `ERB/コマンド関連/COMF/COMF270-299KOJO.ERB`
- `ERB/口上・メッセージ関連/個人口上/...`

其中 `個人口上` 目录下有大量角色专属实现，通常按下面方式拆：

- `イベント`
- `カウンター`
- `日常系コマンド`
- `セクハラコマンド`
- `愛撫系コマンド`
- `奉仕系コマンド`
- `性交系コマンド`
- `刻印取得`
- `絶頂`
- `育児イベント`
- `道具系コマンド`

说明：

- `eraTW` 的口上不是一个文件，而是“通用分发层 + 角色目录 + 子主题文件”的体系。

## 3. eraTW 口上的分层方式

### 3.1 通用层

职责：

- 统一入口
- 路由到当前角色口上
- 处理没有专属口上时的默认行为

### 3.2 角色层

职责：

- 每个角色独有的文案和分支
- 可拆成多个主题文件，避免单文件过大

### 3.3 子主题层

常见切分方式：

- 事件
- 指令
- 反击/Counter
- 刻印取得
- 绝顶
- 特殊事件

## 4. erAL 的迁移原则

### 4.1 前期不复制内容体量

不直接照搬：

- 一开始就为所有角色写专属口上
- 把角色文案拆成几十个文件

因为 `erAL` 当前阶段的瓶颈不是文案容量，而是：

- 核心循环是否稳定
- 指令与状态是否清晰
- 约会、同行、地点、MARK 是否跑顺

### 4.2 前期先做“通用口上层”

第一阶段做法：

1. 所有角色先共用同一套通用文案模板。
2. 模板按指令、地点、时段、关系阶段、约会状态、MARK 变化。
3. 角色只有少量风格标签差异，不做完整专属口上。

### 4.3 后期再做“角色专属覆盖”

第二阶段做法：

1. 挑 1 到 3 名核心角色。
2. 只给这些角色补专属口上。
3. 其余角色继续走通用层。

## 5. 建议的 erAL 口上架构

### 5.1 第一层：通用口上

建议目录：

- `data/base/dialogue.toml`
- 或后续拆成 `data/base/dialogue/common/*.toml`

职责：

- 所有角色兜底文本
- 按 `action_key`
- 按 `required_stage`
- 按 `location_keys`
- 按 `time_slots`
- 按 `requires_date`
- 按 `requires_following`
- 按 `required_marks`

### 5.2 第二层：角色覆盖

建议目录：

- `data/base/characters/<key>/dialogue.toml`
- 后续必要时再拆成：
  - `dialogue_event.toml`
  - `dialogue_daily.toml`
  - `dialogue_date.toml`

职责：

- 只覆盖该角色独有的文本
- 优先级高于通用层

### 5.3 第三层：极少量角色专属子系统

只在后期需要时增加：

- 特定角色的特殊事件
- 特定角色的独有约会链
- 特定角色的誓约后文本

## 6. 推荐的匹配顺序

对 `erAL` 来说，推荐这样匹配：

1. 角色专属事件文本
2. 角色专属指令文本
3. 通用事件文本
4. 通用指令文本
5. 最终兜底文本

也就是：

- 先看“这个角色有没有专属”
- 再看“当前事件有没有通用模板”
- 最后才落到最简单的兜底句

## 7. 推荐的文件拆分节奏

### 阶段 1：现在

- 只保留单文件 `dialogue.toml`
- 先把通用模板补齐
- 少量角色拥有少量专属条目即可

### 阶段 2：系统稳定后

- 为核心角色拆：
  - `日常`
  - `事件`
  - `约会`
  - `誓约后`

### 阶段 3：内容爆发后

- 再考虑进一步按 `日常系 / 亲密系 / 事件系 / Counter / 特殊系` 拆分

## 8. 对 erAL 的直接行动建议

- [x] 前期采用通用口上层
- [ ] 把通用口上单独拆成独立数据层
- [ ] 定义“角色专属覆盖优先于通用层”的匹配顺序
- [ ] 只给少量核心角色补专属口上
- [ ] 后期再考虑细拆成 `事件 / 日常 / 约会 / 誓约后`
