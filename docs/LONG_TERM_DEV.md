# erAL 成品化长期规划

更新时间：2026-04-22

本规划围绕“成品目标”制定，不再使用 MVP 口径。

## 1. 当前阶段定位

当前项目已完成大量系统铺设，进入“表达完整化 + 稳定化 + 内容规模化”阶段。

阶段目标顺序：

1. 先把已做系统完整表达给玩家。
2. 再做稳定性和可维护性加固。
3. 最后扩大内容体量与表现层质量。

## 2. 里程碑框架（Product Track）

### P1 表达完整化（当前）

目标：系统存在即有玩家感知。

交付物：

1. 系统到玩家反馈的全量映射表。
2. 关键轴（TALENT/PALAM/标记）减少“只写不读”。
3. 指令失败反馈、事件反馈、状态反馈统一可感知。

完成判据：

1. 每个已实现系统至少一个玩家可见反馈入口。
2. 新增字段必须绑定消费者与测试。
3. 文档口径与运行行为一致。

### P2 稳定性与质量门

目标：可持续迭代，不靠手工排雷。

交付物：

1. 核心链路回归集（指令-结算-事件-口上-存档）。
2. 内容坏样例夹具与校验规则增强。
3. 回归日志分层索引（系统维度定位失败）。

完成判据：

1. 核心链路回归稳定。
2. 内容错误能在进入主分支前被发现。
3. 高风险模块有可重复验证路径。

### P3 内容工业化

目标：高密度内容可批量生产。

交付物：

1. 角色包分级模板（最小包/标准包/高密包）。
2. 口上与事件批量工具链。
3. 内容密度与触发覆盖指标。

完成判据：

1. 新角色接入时间和质量可预测。
2. 角色间内容深度差异可被量化管理。
3. 大规模扩容不需要频繁改核心代码。

### P4 体验打磨与发布化

目标：形成可持续发布的产品工程。

交付物：

1. 版本化存档兼容策略。
2. 发版检查清单与回滚策略。
3. 运行日志与关键行为统计口径。

完成判据：

1. 每次版本发布有明确质量报告。
2. 存档兼容风险可控。
3. 线上问题可回溯、可定位、可修复。

## 3. 中长期能力地图

### 3.1 系统层

1. 持续补齐 TALENT/PALAM 消费端。
2. 扩充事件条件体系与长期记忆系统联动。
3. 强化调教链路的可测性和可解释性。

### 3.2 内容层

1. 扩展角色包数量与阶段差分密度。
2. 扩展地点功能化设计与区域叙事。
3. 完善节日/季节/天气与角色行为联动。

### 3.3 表现层

1. 维持现有可运行 UI 的连续迭代，不空转重写。
2. 强化状态面板、地图信息、反馈一致性。
3. 逐步接入音效、立绘与更强表现能力。

## 4. 执行策略

1. 先补闭环，再扩系统。
2. 先补消费者，再扩数据。
3. 先补测试，再做大规模内容导入。
4. 每个里程碑都要产出“玩家可感知变化”而不只是代码变化。

## 5. 风险与对策

1. 风险：文档与实现再次漂移。
   对策：A 层文档每周核对一次，变更后立即同步。

2. 风险：内容扩张导致回归激增。
   对策：先做质量门，再提内容产能。

3. 风险：数值轴持续堆积但无消费者。
   对策：新增轴规则必须附带消费者与测试。

4. 风险：阶段目标被“新功能冲动”打散。
   对策：按 P1-P4 严格执行，不跨阶段插队。
