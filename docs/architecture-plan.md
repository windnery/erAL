# erAL 架构规划与审查

## 1. 结论先说

你现在的目录骨架：

```text
erAL/
  build/
  core/
  data/
  editor/
  image/
  mod/
  scripts/
```

不能算错，但只适合“刚开工程”的阶段，不适合直接长期扩展成一个大型 era 系项目。

主要问题不在于目录名本身，而在于边界还不够清晰：

- `core/` 过大，后续很容易把“运行时基础设施”“玩法规则”“角色行为”“地图逻辑”“UI 组装”全部塞进去。
- `data/` 和 `mod/` 的职责重叠，未来会很难区分“官方内容”“编译后的生成数据”“模组覆盖数据”。
- `scripts/` 容易沦为杂物间，把构建、导表、校验、迁移、打包都混在一起。
- `image/` 只覆盖图片资源，不足以表达完整资产层；后面音频、字体、局部 UI 资源、角色差分都会继续长出来。
- 缺少 `docs/`、`tests/`、`runtime/` 或 `saves/` 这类工程必需层，说明现在更像“想法目录”，还不是“可演进架构”。

所以我的判断是：

- 方向是合理的，你已经自然分出了“核心”“数据”“编辑器”“模组”“资源”。
- 但粒度太粗，不适合承载你这个目标项目。
- 现在正是改的最佳时机，因为还没产生迁移成本。

## 2. 你的项目目标决定了什么架构

你的项目不是单纯复刻 eraTW，也不是单纯学习 erArk。

它实际上有三个同时成立的目标：

1. 吸收 eraTW 的玩法语义。
2. 借鉴 erArk 的 Python 工程经验。
3. 以碧蓝航线港区后宫为核心，做一个长期可扩展的新项目。

这意味着架构不能直接走两种极端：

- 不能做成纯 era 脚本式堆叠。
- 也不能做成纯通用引擎，把玩法语义抽得过度抽象。

更合适的方向是：

- 用 Python 做“运行时和工具链”。
- 用数据驱动承载大多数玩法配置。
- 用少量领域脚本或规则函数承载高变动内容。
- 把“基础引擎”和“港区领域规则”严格分层。

一句话概括就是：

> 用 erArk 的工程化手段，承接 eraTW 的玩法组织方式，但把“港区日常经营 + 舰娘关系推进 + 事件口上生产”作为第一优先级来定边界。

## 3. 从 eraTW 应该学什么，不该学什么

### 应该学的

从本地 `eraTW` 结构看，它最值得学的不是语法，而是“玩法语义如何分层”：

- `CSV/` 负责全局枚举、参数轴、角色基础表、指令表。
- `CSV/Chara/` 负责角色内容数据拆分。
- `ERB/コマンド関連`、`ERB/イベント関連`、`ERB/MOVEMENTS`、`ERB/ステータス計算関連` 体现了玩法子系统分工。
- `ERB/口上・メッセージ関連/個人口上` 体现了角色内容可以大规模拆包扩展。
- `COMMON.ERB`、`DIM.ERH` 这类公共层负责“共享变量语义”和“全局函数库”。

它真正强的地方是：

- 玩法维度非常多，但大多通过统一的枚举和状态语义组织起来。
- 角色内容虽然巨大，但可以按角色、按指令类别、按事件类别拆分。
- 大量系统虽然耦合高，但作者群体依然能持续加内容，因为内容入口是稳定的。

### 不该学的

不要继承 eraTW 这些问题：

- 全局变量过多，语义要靠命名和约定记忆。
- 数值索引驱动过强，类型信息弱。
- 系统之间的调用链隐式且脆弱。
- “玩法逻辑”和“文本表现”经常混在一起。
- 很多内容扩展依赖对既有全局状态的经验性理解，新作者门槛高。

对 `erAL` 来说，最应该避免的是：

> 把 `CFLAG/TFLAG/ABL/TALENT` 这套语义原样搬成一堆匿名整数数组。

正确做法应该是：

- 保留 era 系的语义分类。
- 但在 Python 层把它们变成可读、可校验、可序列化的结构化模型。

## 4. 从 erArk 应该学什么，不该学什么

`erArk` 的公开仓库可以看出比较清楚的 Python 分层思路：

- `Script/Config` 放配置定义，如角色、地图、名字、通用配置。
- `Script/Core` 放基础能力，如常量、路径、初始化、流程处理、图像、算法。
- `Script/Design` 放领域对象和玩法设计，如角色、属性计算、行为、移动。
- `Script/StateMachine` 放流程状态机。
- `Script/System` 放大系统，如指令、医疗、性系统、委托系统。
- `Script/UI` 放界面流、模块、面板。

这套分法的优点非常明确：

- 配置、基础设施、领域逻辑、系统、UI 有分层意识。
- 适合 Python 项目长期演进。
- 对工具链、编辑器、Web/TK 双前端等扩展更友好。

但它也有一个你需要提前规避的问题：

- `Design/` 和 `System/` 的边界如果不继续收紧，后期容易出现“双核心”。

也就是：

- `Design` 开始写领域对象和规则。
- `System` 又写一遍系统驱动和规则编排。
- 最后角色逻辑、事件逻辑、结算逻辑在两边来回漂移。

对你的项目，这会更危险，因为你还要兼顾：

- 港区经营
- 地图/位置
- 日程推进
- 角色关系
- 事件触发
- 口上文本
- H 指令和日常指令
- 模组内容

如果边界不够清楚，后面一定会变成到处都能改状态。

## 5. 适合 erAL 的推荐架构

我建议你把工程改成“六层结构”。

```text
erAL/
  docs/                 # 架构、数据规范、玩法设计文档
  src/
    eral/
      app/              # 启动、依赖装配、配置读取、生命周期
      engine/           # 与具体题材无关的运行时基础设施
      domain/           # 港区题材直接相关的领域模型
      systems/          # 玩法系统编排与结算
      content/          # 官方内容入口、内容注册、模板
      ui/               # 前端适配层（CLI/TUI/Web/Editor API）
      tools/            # 导表、编译、校验、迁移
  data/
    base/               # 官方静态原始数据
    generated/          # 编译生成数据，不手改
    locales/            # 本地化文本
  mods/
    official/           # 官方分包内容
    user/               # 用户模组
  assets/
    image/
    audio/
    fonts/
  tests/
  scripts/
  runtime/
    saves/
    cache/
    logs/
```

### 各层职责

#### `app/`

只做装配，不写玩法。

负责：

- 启动入口
- 配置加载
- 存档路径决定
- 注册系统
- 初始化内容仓库
- 组装 UI 和游戏循环

#### `engine/`

这里必须是“题材无关”的。

负责：

- 时间线与 tick 推进
- 事件总线
- 指令调度器
- 条件表达式执行
- 存档序列化
- 资源定位
- 数据注册表
- 随机数服务
- 地图图结构、寻路等通用能力

不要把“舰娘好感”“港区设施”“秘书舰逻辑”写进这里。

#### `domain/`

这里放“碧蓝航线港区后宫”这个题材本身的核心对象。

建议至少拆出：

- `domain.character`
- `domain.player`
- `domain.port`
- `domain.room`
- `domain.schedule`
- `domain.relationship`
- `domain.inventory`
- `domain.quest` 或 `domain.commission`
- `domain.scene`

这里的目标是：

- 定义状态结构。
- 定义领域不变量。
- 不直接处理 UI。

例如：

- 角色有哪些基础属性、成长属性、情绪、欲情、依赖、恋慕、服从。
- 港区有哪些地点、房间、设施、开放条件。
- 一天被分成哪些时段。
- 玩家与角色的关系如何分段。

#### `systems/`

这里是玩法系统层，不是数据定义层。

建议拆成：

- `systems.time`
- `systems.map`
- `systems.command`
- `systems.relationship`
- `systems.daily`
- `systems.event`
- `systems.dialogue`
- `systems.h_scene`
- `systems.commission`
- `systems.facility`
- `systems.ai`
- `systems.save`

系统层负责：

- 在一个时点读取状态。
- 执行规则。
- 产出状态变更和表现事件。

一句话说：

- `domain` 定义“世界是什么”。
- `systems` 定义“世界怎样变化”。

#### `content/`

这是你最不能省的一层。

因为你的项目最终一定会被大量角色内容和事件内容淹没。

这里要负责：

- 角色内容注册
- 指令定义注册
- 事件模板注册
- 文本片段与口上入口
- 官方内容包管理

后期你甚至可以把“角色包”做成标准结构：

```text
content/characters/enterprise/
  character.yaml
  stats.yaml
  schedules.yaml
  events.yaml
  dialogue/
  h/
  images/
```

这会比把所有东西直接塞到 `data/` 或 `mod/` 强得多。

#### `ui/`

这里只放表现和交互，不放规则。

不管你以后做：

- CLI
- TUI
- Web
- 桌面编辑器

都应该从 `systems` 获取结果，从 `content` 取文本和资源，而不是在 UI 里直接改核心状态。

## 7. 数据建模建议

这是整套架构里最关键的部分。

### 7.1 保留 era 语义，但不要保留 era 的裸数组写法

建议把 era 常见语义改造成结构化模型：

- `Base` -> 基础体能/状态类字段
- `Abl` -> 能力等级
- `Talent` -> 特质集合
- `Flag` -> 全局持久状态
- `CFlag` -> 角色持久状态
- `TFlag` -> 临时流程状态
- `Mark` -> 烙印/阶段性状态
- `Palam` -> 可累积经验或刺激值
- `Source` -> 一次行为产生的增量

其中最重要的一条是：

- `Source` 和 `结算` 要分离。

也就是行为先产生“变化提案”，再统一结算，不要每个指令直接把状态改到底。

这样你才能稳定处理：

- 天赋修正
- 心情修正
- 场景修正
- 服装修正
- 房间修正
- 昼夜修正
- 首次/重复/关系阶段修正

### 7.2 推荐的数据流

```text
玩家输入指令
-> command system 解析可执行项
-> premise checker 验证前提
-> scene builder 收集上下文
-> action 产出 source/effect
-> settle pipeline 统一结算
-> event system 检查是否触发派生事件
-> dialogue system 选择文本
-> ui 渲染结果
```

这条链路很重要，因为它正好把 eraTW 的“指令 -> 增量 -> 结算 -> 口上”抽成了 Python 可维护流程。

### 7.3 角色内容不要直接写 Python 业务代码

建议分三级：

1. 结构化数据
2. 条件表达式
3. 少量 Python 特例处理

也就是：

- 大多数角色属性、日程、事件条件、文本入口放 YAML/JSON/TOML。
- 条件判断尽量用你自己的小表达式系统或注册条件函数。
- 只有极少数特殊角色机制再写 Python 代码。

这样编辑器、模组、校验器才能真正成立。

## 8. 面向“港区后宫”的专项建议

这部分是你和 eraTW、erArk 最大的差异点，所以架构必须提前为它让路。

### 8.1 把“地点”当成一等公民

港区题材和传统 era 最大区别之一，是“空间和同居关系”更重要。

所以你要尽早确立：

- 港区地图
- 房间/宿舍/办公室/澡堂/食堂/训练场/码头/指挥室
- 角色所在位置
- 可见角色集合
- 角色是否可被约见、尾随、夜袭、同宿、值班

这意味着地图和房间系统不能只是附属功能，应该是核心系统。

### 8.2 把“日程”当成骨架系统

建议一天先切成有限时段，而不是一开始就用分钟级全模拟。

比如：

- 清晨
- 上午
- 午后
- 傍晚
- 夜晚
- 深夜

优点是：

- 事件调度容易控制
- AI 日程容易写
- 文本成本可控
- 存档状态更稳定

等系统成熟后，再把局部玩法细化到更小时间粒度。

### 8.3 把“关系层级”独立成系统

不要把好感、依赖、服从、恋慕、嫉妒、独占欲全部散在角色字段里临时判。

建议至少单独维护：

- `affection`
- `trust`
- `desire`
- `obedience`
- `intimacy_stage`
- `exclusive_state`

系统里再定义：

- 解锁条件
- 事件门槛
- 口上风格切换
- 指令可用性

### 8.4 正常指令和 H 指令不要共用同一层文本组织

它们可以共用指令框架，但内容包结构最好分开。

因为：

- 产量不同
- 触发条件不同
- 前提校验不同
- 文本写法不同
- 版本管理不同

推荐：

- 日常系统和日常口上独立目录
- H 系统和 H 文本独立目录
- 统一只在 `command registry` 里汇总

## 9. 推荐的开发阶段

### 阶段 0：重整骨架

目标：

- 先把目录边界立住。
- 不急着做大量玩法。

交付物：

- 新目录结构
- 基础配置加载
- 日志
- 存档目录
- 数据注册表

### 阶段 1：最小可玩循环

目标：

- 做出“港区一天循环”。

最小集合：

- 地点切换
- 时段推进
- 角色位置刷新
- 见面/对话/送礼/休息/邀请同行
- 基础关系变化
- 基础事件触发

### 阶段 2：角色内容管线

目标：

- 让新角色可以低成本接入。

交付物：

- 角色数据 schema
- 事件 schema
- 文本 schema
- 校验器
- 导入器
- 编辑器对接格式

### 阶段 3：H 系统与特殊事件

目标：

- 在稳定关系和场景框架下接入 H 流程。

交付物：

- H 指令前提系统
- Source/Settle 统一结算
- 场景上下文标签
- H 文本选择器

### 阶段 4：设施、委托、养成扩展

目标：

- 把港区经营层补齐。

交付物：

- 委托
- 宿舍/房间升级
- 礼物/收藏
- 特殊节日
- 阵营或舰种差分机制

### 阶段 5：编辑器和模组生态

目标：

- 降低内容生产门槛。

交付物：

- 编辑器数据协议
- 模组加载顺序
- 冲突覆盖规则
- 内容校验报告

## 10. 我给你的最终建议

### 当前架构是否合理

结论是：

- 合理，但只合理到“立项初期”。
- 如果继续按现在的粗目录直接写，半年后一定会出现系统边界混乱。

### 最值得马上改的三件事

1. 不要再把所有核心都压进 `core/`。
2. 尽快把“engine / domain / systems / content / ui”五层拆开。
3. 提前定义“角色数据、指令数据、事件数据、文本数据”的 schema 和编译流程。

### 最应该坚持的三条原则

1. 状态结构化，拒绝匿名大数组。
2. 指令先产出增量，再统一结算。
3. 文本内容和规则逻辑分离，但通过稳定上下文绑定。
