# erAL 长期开发指导书（AI Vibe Coding，可靠性优先）

## 1. 文档目的

这份文档用于指导 AI 和开发者在长期项目中协作，确保：

1. 方向稳定，不因短期灵感频繁漂移。
2. 质量可控，任何新增内容都可验证、可回归、可追溯。
3. 可持续扩展，角色包和玩法扩展不依赖改动核心系统。

适用范围：`erAL` 全仓（系统代码、内容数据、测试、工具链、文档）。

## 2. 顶层原则（必须遵守）

1. 语义继承优先：优先继承 eraTW 成熟语义，工程实现可重做。
2. 可靠性优先：先保证正确性和可维护性，再追求短期可玩观感。
3. 单一事实源：验收标准仅以 `docs/PRD.md` 为准，任务状态仅以 `docs/TODO.md` 为准。
4. 数据驱动优先：规则和内容优先外置到 TOML/JSON；系统层不硬编码剧情。
5. 小步可回滚：每次改动可独立验证，失败可快速定位与回退。

## 3. 文档治理模型

### 3.1 文档分工

1. `docs/PRD.md`：产品目标、范围边界、验收定义。
2. `docs/TODO.md`：任务看板与推进状态。
3. `docs/COMMAND_MIGRATION.md`：指令迁移队列与迁移状态。
4. `docs/architecture-plan.md`：架构约束、演进计划与边界说明。
5. 本文档：AI 协作协议、质量门、长期推进机制。

### 3.2 更新规则

1. 任何影响范围、验收、阶段定义的变更，必须先改 `PRD.md`。
2. 任何实现任务，必须在 `TODO.md` 有任务卡后再开工。
3. 任何新增或调整指令，必须同步更新 `COMMAND_MIGRATION.md`。
4. 任何破坏性架构改动，必须先写 ADR（见 11.2）。

## 4. 生命周期与里程碑（长期项目版）

### 4.1 里程碑

1. `L0 架构与质量基线`
   - 六层边界稳定
   - 主干回归测试稳定
   - 内容加载、校验、存档流程稳定

2. `L1 核心语义完整`
   - SOURCE 结算链、关系阶段、MARK 分支规则一致
   - 关键指令迁移机制稳定
   - 关键状态字段定义与测试覆盖稳定

3. `L2 内容规模化生产`
   - 角色包模板成熟
   - 内容校验器可发现结构缺陷与数量缺口
   - 角色新增流程无需改核心系统

4. `L3 MVP`
   - 满足 PRD 的数量目标与连续游玩目标
   - 文本、事件、指令与地图联动体验达标

### 4.2 优先级规则（强制）

优先级严格按照以下顺序执行：

1. P0：可靠性与一致性（测试、校验、文档口径、关键 BUG）
2. P1：架构与工具能力（可观测性、自动化、内容管线）
3. P2：内容扩展与体感优化

说明：不允许以“短期可玩效果”跳过 P0/P1。

## 5. 任务卡规范（AI 执行格式）

所有任务必须包含以下字段：

- 标题（动词开头）
- type（code/content/test/docs/tooling）
- priority（P0/P1/P2）
- milestone（L0/L1/L2/L3）
- est（25m/45m/90m）
- DoD（完成定义）
- verify（验证命令或步骤）
- owner（human/ai/pair）
- status（todo/doing/blocked/done）
- updated（日期）

任务若缺任何字段，视为不可执行。

## 6. AI 会话协议（长期协作版）

### 6.1 会话输入模板

1. 本次目标（一个主目标）
2. 范围边界（明确不做什么）
3. 验收标准（可测量）
4. 禁止改动文件
5. 输出要求（代码/测试/文档/报告）

### 6.2 会话输出模板

1. 完成项
2. 变更文件列表
3. 验证结果
4. 风险与遗留
5. 下一步最小任务（<=90m）

### 6.3 AI 默认行为

0. 先读 `docs/README.md`，确认文档级别与权威边界，再进入具体文档。
1. 先读相关文档再改代码。
2. 每次改动至少附一条验证动作。
3. 发现范围膨胀时，拆成 Backlog 任务而非继续扩写。
4. 若发现文档与实现冲突，先记录并修正文档口径，再推进实现。

## 7. 质量门（Quality Gates）

### 7.1 开发前质量门

1. 任务卡字段完整。
2. 影响范围明确（涉及系统、内容、测试、文档哪些层）。
3. 验证方式明确（自动化/手工）。

### 7.2 提交前质量门

1. 相关测试通过。
2. 内容校验通过。
3. 文档同步更新。
4. 没有引入新的已知错误。

### 7.3 里程碑质量门

1. 关键回归测试稳定。
2. 关键指标达标（见第 10 节）。
3. 无高优先级阻塞项。

## 8. 测试与验证策略

### 8.1 测试层级

1. 单元测试：核心规则与数据转换。
2. 集成测试：系统协作链路（指令 -> 结算 -> 事件 -> 对话）。
3. 烟测：连续多时段/多天运行稳定性。
4. 回归测试：已修复 BUG 的复现与防回退测试。

### 8.2 必跑命令

在 `erAL` 目录执行：

1. `python -m unittest discover -s tests -t .`
2. `python -m eral.tools.validate_content --root .`

若任务影响坐标轴或兼容轴导入，再执行：

3. `python -m eral.tools.import_tw_axes --source "../eratw4.980-260320-SUNNY(PC)/CSV" --output data/generated/tw_axis_registry.json`

## 9. 可观测性与故障处理

### 9.1 最小日志规范

日志至少包含：

1. 日期与时段
2. 玩家动作与指令 key
3. 目标角色 key
4. 命中事件 key
5. 失败原因（若动作失败）

### 9.2 BUG 处理流程

1. 记录现象与版本。
2. 给出最小复现步骤。
3. 附带可复现存档（若有）。
4. 修复后新增回归测试。
5. 在周报中登记该问题关闭。

## 10. 长期健康指标（每周更新）

1. 测试通过率（全量）
2. 新增回归测试数量
3. 内容校验失败数
4. 高优先级缺陷存量
5. 每周新增事件数/对话数/指令数
6. 文档一致性问题数量

## 11. 变更治理

### 11.1 何时必须走 RFC

满足任一条件必须先写 RFC：

1. 改变核心数据模型（如关系阶段、兼容轴结构）
2. 改变结算主链路
3. 影响 3 个以上系统模块的改动

### 11.2 何时必须写 ADR

满足任一条件必须新增 ADR：

1. 新增长期架构约束
2. 放弃既有实现路线
3. 引入新的跨模块基础设施

ADR 建议存放于 `docs/architecture/adr/`，命名格式：`ADR-YYYYMMDD-主题.md`。

## 12. 角色模板策略（可靠性版）

采用两层模板，避免一次性全量扩字段造成不可控：

1. 基线模板（L1）
   - 最小必要字段：`base/palam + cflag(2/4/6) + 少量 talent/abl + marks`
   - 目标：保证规则链条稳定和分支可测

2. 扩展模板（L2）
   - 逐步补齐 eraTW 兼容字段
   - 条件：每新增字段必须绑定可感知行为差异或规则价值

## 13. 给 AI 的禁止事项

1. 不在未定义任务卡时直接改核心系统。
2. 不把“想法”直接写成大范围代码改动。
3. 不跳过测试和内容校验直接宣告完成。
4. 不在同一会话同时推进多个跨模块重构。

## 14. 周期化执行节奏

1. 每周一：确定本周 P0 任务与风险。
2. 每周三：中期检查（测试稳定性、阻塞项处理）。
3. 每周五：复盘（指标、风险、下周计划）。

## 15. 与 TODO 的关系

执行规则：

1. 本文档定义“如何做”。
2. `docs/TODO.md` 定义“做什么”。
3. 若二者冲突，以本文档的质量门与治理规则优先，再修订 TODO。
