## 2. 当前架构状态

### 2.1 已冻结的数据契约

1. `data/base/axes/*` 是当前数值轴真相。
2. `data/base/commands/train.toml` 是唯一命令来源。
3. `data/base/effects/command_effects.toml` 是声明式效果入口之一，但目前内容未补完。
4. 运行时以 index 为主，不再引入新的语义字符串 key 作为正式契约。
5. `CFLAG` 保留；`FLAG / TFLAG` 不再作为新运行时设计目标。

### 2.2 当前真实代码状态

1. 新数据契约已进入：
   `axes / train / command_effects`
2. 旧 Python 逻辑仍大量存在于：
   `content / systems / ui / specs`
3. 当前阶段的主要矛盾不是“缺系统”，而是“新数据契约和旧执行逻辑并存”。

## 4. 当前核心缺口

### 4.1 指令执行模型尚未完成收束

现状：

1. `train.toml` 已是唯一命令来源。
2. `CommandService` 仍承接了大量旧时代逻辑。
3. `command_effects.toml` 只覆盖了一部分 SOURCE 产出。

结果：

1. 指令驱动理念已确立，但运行时模型还没有完全按它重写。

### 4.2 指令字段边界尚未正式冻结

现状：

1. 哪些字段属于命令定义层。
2. 哪些字段属于声明式效果层。
3. 哪些内容必须留在 Python operation/gate 层。

这些边界还没完全变成代码约束。

### 4.3 specs 文档大面积滞后

现状：

1. 多份 `specs` 仍引用 `commands.toml`。
2. 部分 spec 假设旧数值模型和旧命令来源仍然有效。

结论：

1. 当前 `specs` 只能当参考，不能当执行真相。
