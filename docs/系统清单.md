# erAL 系统清单（Product Baseline）

更新时间：2026-04-22

本文档只记录“当前仓库真实存在且可运行”的系统，不记录历史计划口径。

## 1. 规模快照

1. 系统模块：35（src/eral/systems）
2. 指令条目：162（data/base/commands.toml）
3. 角色包：5（akashi / enterprise / javelin / laffey / shiranui）
4. 事件条目：97（全局 + 角色包）
5. 口上条目：295（全局 + 角色包）
6. 测试文件：62（tests）
7. 测试函数：632（按 def test_ 统计）

## 2. 系统分组总览

### 2.1 主链编排层

1. 命令执行：commands + command_gates + settlement
2. 场景与反馈：scene + events + dialogue
3. 回合推进：game_loop + time_service

状态：已贯通，可运行。

### 2.2 角色与关系层

1. relationships（阶段推进）
2. companions / dates（同行与约会状态）
3. marks / imprint（长期状态变化）

状态：主能力可用，部分深层行为仍待补强。

### 2.3 地图与世界层

1. navigation（路径与移动）
2. schedule（日程刷新）
3. distribution（动态分布）
4. weather + calendar + ambient_events（环境变化）

状态：有完整基础链路，适合继续做内容扩容。

### 2.4 数值与成长层

1. vital + fatigue（体力气力）
2. favor_calc（关系增长公式）
3. source_extra（天赋修正）
4. abl_upgrade + palam_decay

状态：计算骨架可用，消费者密度仍不均衡。

### 2.5 经济与资产层

1. wallet（双账户资金）
2. shop + gifts（购买与送礼）
3. facilities + commissions（长期循环）
4. skins（外观）

状态：系统链路完整，内容密度可继续提升。

### 2.6 特化玩法层

1. training（调教主系统）
2. ejaculation（兴奋与液体）
3. persistent state（持续状态）

状态：功能具备，需持续加固可测性与反馈一致性。

### 2.7 基础设施层

1. save（存档读档）
2. content loaders（静态数据加载）
3. validate_content / import_tw_axes（工具链）

状态：可支撑持续迭代。

## 3. 已实现系统的玩家可感知覆盖

覆盖等级定义：

1. A：玩家在单次游玩中可直接感知输入和反馈。
2. B：玩家可间接感知，但反馈强度较弱。
3. C：当前主要在数据或显示层，反馈不足。

当前评估：

1. A 级：命令主链、关系推进、事件口上、地图移动、时间推进、存档、经济购买。
2. B 级：天气、随机氛围事件、设施加成、部分持续状态。
3. C 级：部分 TALENT/PALAM 的真实逻辑消费者仍偏少。

## 4. 当前主要缺口（按成品目标）

1. 一些数值轴“写入多、消费少”，导致玩家难以感知差异。
2. 部分高复杂模块测试密度不足，内容扩张后回归风险高。
3. 文档与运行入口、表现层现实之间曾存在口径偏差，已开始修复但需持续巡检。

## 5. 本阶段补强优先级

1. P0：补齐关键 TALENT/PALAM 消费端。
2. P0：主链高风险回归集（命令-结算-事件-口上-存档）。
3. P1：反馈一致性（失败原因、状态变化、事件提示）。
4. P1：内容校验坏样例与工具链增强。

## 6. 文档同步规则

1. 系统新增或下线时，先改代码，再改本清单。
2. 数据规模变化（指令数、角色包数、事件口上数）需同步刷新。
3. 对玩家可感知覆盖等级有变化时，必须同步更新 `docs/玩家体验映射.md`。
