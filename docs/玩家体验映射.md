# erAL 玩家体验映射（V1）

更新时间：2026-04-22

目的：把“已经做完的系统”转换成“玩家能感受到的结果”，用于成品化阶段验收。

## 1. 核心玩法主链

链路：玩家输入 -> 指令门禁 -> SOURCE 产出 -> 结算 -> 事件匹配 -> 口上选择 -> 时间推进

玩家可感知点：

1. 同样动作在不同地点、时段、关系阶段会出现不同可执行结果。
2. 执行动作后，角色态度会连续变化，不是一次性跳变。
3. 命中事件时文本明显不同；没命中事件也有动作兜底文本。
4. 下一时段角色位置会变化，玩家需要主动找人。

## 2. 已落地系统到玩家反馈映射

### 2.1 指令与门禁

- 系统：CommandService + command_gates + commands.toml
- 玩家反馈：
  - 可执行指令会随地点/时段/阶段动态变化。
  - 不可执行时有失败原因，不是静默失败。

### 2.2 结算与关系

- 系统：SettlementService + RelationshipService + relationship_stages.toml
- 玩家反馈：
  - 同样互动反复进行后，会从陌生推进到更高关系阶段。
  - 阶段提升会带来新互动、新事件和新口上。

### 2.3 事件与口上

- 系统：EventService + DialogueService + events/dialogue 数据包
- 玩家反馈：
  - 特定角色、地点、时段下会命中专属事件。
  - 即使事件未命中，也能得到符合当前行为的文本反馈。

### 2.4 地图、移动、日程、分布

- 系统：NavigationService + ScheduleService + DistributionService + port_map
- 玩家反馈：
  - 地图不是装饰，移动会消耗行动资源并影响可互动对象。
  - 角色不会原地不动，找人是持续性的玩法动作。

### 2.5 时间、天气、随机事件

- 系统：TimeService + CalendarService + WeatherService + AmbientEventService
- 玩家反馈：
  - 日/时段持续推进，环境状态变化会影响场景氛围。
  - 存在非主线随机氛围事件，提升世界活性。

### 2.6 体力与恢复

- 系统：VitalService + fatigue + downbase
- 玩家反馈：
  - 连续高强度行动会累，休息与时段推进能恢复。
  - 玩家需要安排节奏，而不是无限连点。

### 2.7 经济、商店、设施、委托

- 系统：WalletService + ShopService + FacilityService + CommissionService
- 玩家反馈：
  - 资金获取、消费、设施升级形成长期循环。
  - 商店和道具不仅是展示，而是可进入可购买可影响后续行为。

### 2.8 皮肤与外观

- 系统：SkinService + skins/appearances + 商店入口
- 玩家反馈：
  - 角色外观可被获取与切换，部分互动文本可反映外观状态。

### 2.9 调教扩展链

- 系统：TrainingService + ImprintService + EjaculationService + persistent states
- 玩家反馈：
  - 调教存在可持续状态、反击/主动行为与结果态差分。
  - 不是单次指令孤立反馈，而是跨回合持续影响。

### 2.10 存档与跨回合连续性

- 系统：SaveService + runtime/saves
- 玩家反馈：
  - 退出后可继续，关系、状态、进度不会丢失。

## 3. V1 体验覆盖检查表

每个迭代至少通过以下检查：

1. 任意新增系统必须能指出至少 1 个玩家可见反馈。
2. 任意新增数据字段必须能指出至少 1 个运行时消费者。
3. 任意新增指令必须具备：门禁结果、数值变化、文本反馈三件套。
4. 任意新增角色包必须具备：日程、事件、口上、基础互动闭环。
