# 季节与节日规则接入

## 概述

季节与节日规则接入负责让已经存在的时间与日历系统真正进入玩法消费层。当前第一批真实消费者是：

- `skin_shop` 的上架规则
- 事件与口上的时间语义分支

本系统的目标不是实现完整的活动框架，而是先让：

- `season`
- `festival_tags`

这两类高层时间语义，稳定地进入商店、事件和口上判定。

## 当前阶段定位

- 以 `skin_shop` 为第一消费者，优先验证季节/节日标签能影响商品可见性。
- 事件与口上作为第二消费者，复用同一套时间语义条件接口。
- 当前阶段只做最小规则接入，不做活动 DSL，也不实现复杂日期窗口。

## 数据结构

### Skin Seasonal Visibility

在现有 `SkinDefinition` 上新增：

| 字段 | 类型 | 说明 |
|------|------|------|
| `season_visibility` | str | 季节上架规则，如 `always` / `summer` / `winter` |
| `festival_visibility` | [str] | 节日标签要求，如 `["summer_festival"]` |

说明：

- `grant_mode != "shop"` 的皮肤不参与 `skin_shop` 上架判定。
- `season_visibility` 与 `festival_visibility` 都是附加过滤器，不替代 `grant_mode` 与 `shop_visibility`。

### Dialogue Seasonal Conditions

在 `DialogueEntry` 上新增：

| 字段 | 类型 | 说明 |
|------|------|------|
| `required_season` | str? | 需要的季节，如 `summer` |
| `required_festival_tags` | [str] | 需要命中的节日标签 |

### Event Seasonal Conditions

在 `EventDefinition` 上新增：

| 字段 | 类型 | 说明 |
|------|------|------|
| `required_season` | str? | 需要的季节 |
| `required_festival_tags` | [str] | 需要命中的节日标签 |

### Scene Time Semantics

场景上下文应继续扩展并暴露高层时间语义：

| 字段 | 类型 | 说明 |
|------|------|------|
| `season` | str | 当前季节 |
| `festival_tags` | [str] | 当天节日标签 |

## 核心规则

### 高层语义先于底层日期

当前阶段内容层只消费：

- `season`
- `festival_tags`

而不直接消费：

- `month`
- `day`
- `weekday`

原因：

- 高层语义更稳定
- 内容作者更容易理解
- 后续活动规则可统一从时间系统派生

### `skin_shop` 是第一消费者

皮肤上架判定优先顺序建议固定为：

1. `grant_mode`
   - 非 `shop` 的皮肤不进入商店

2. `shop_visibility`
   - `always`：允许继续判断
   - `manual`：默认不上架

3. `season_visibility`
   - 若不是 `always`，要求当前 `season` 匹配

4. `festival_visibility`
   - 若非空，要求当前 `festival_tags` 至少命中一项

### 事件与口上是第二消费者

事件与口上复用同一套时间语义：

- `required_season`
- `required_festival_tags`

这样可以保证：

- 皮肤商店和文本系统读取的是同一套时间标签
- 不会出现“商店按夏季上架，但口上另写一套月份判断”的分裂

## 流程

### 皮肤上架判定

1. `skin_shop` 打开
2. `SkinRuleEvaluator` 读取当前世界时间语义：
   - `season`
   - `festival_tags`
3. 逐个评估 `SkinDefinition`
4. 只有满足全部可见性规则的皮肤才出现在商店里

### 事件与口上判定

1. `SceneService` 构建 `SceneContext`
2. 将 `season` 与 `festival_tags` 写入上下文
3. `EventService` 与 `DialogueService` 在原有条件上追加时间语义匹配
4. 当时间条件满足时，命中特定季节或节日内容

## 配置

### skins.toml

示例：

```toml
[[skins]]
key = "enterprise_summer"
actor_key = "enterprise"
display_name = "夏日泳装"
price = 1680
grant_mode = "shop"
shop_visibility = "always"
season_visibility = "summer"
festival_visibility = []
tags = ["summer", "swimsuit"]
appearance_key = "enterprise_summer"

[[skins]]
key = "enterprise_festival"
actor_key = "enterprise"
display_name = "夏日祭礼服"
price = 1880
grant_mode = "shop"
shop_visibility = "always"
season_visibility = "summer"
festival_visibility = ["summer_festival"]
tags = ["summer", "festival", "formal"]
appearance_key = "enterprise_festival"
```

### dialogue.toml / events.toml

示例：

```toml
[[entries]]
key = "chat"
actor_key = "enterprise"
priority = 7
required_season = "summer"
lines = ["「海风的味道和平时不一样……夏天确实到了。」"]

[[entries]]
key = "chat"
actor_key = "enterprise"
priority = 8
required_festival_tags = ["summer_festival"]
lines = ["「夏日祭的灯光很漂亮。今天就别急着回去了。」"]
```

## 与现有系统的关系

- 依赖 [时间与日历系统](时间与日历系统.md)
  - `season` 与 `festival_tags` 的唯一来源
- 依赖 [皮肤与外观系统](皮肤与外观系统.md)
  - `SkinRuleEvaluator` 负责消费时间语义并决定皮肤上架
- 依赖 [事件系统](事件系统.md)
  - 时间语义成为事件匹配条件
- 依赖 [口上系统](口上系统.md)
  - 时间语义成为对话分支条件
- 依赖 [商店与经济系统](商店与经济系统.md)
  - 皮肤商店首次真正使用时间标签驱动商品可见性

## 当前阶段不做

- 不做完整活动系统
- 不做复杂日期窗口或年份限定
- 不做天气与节日的联动效果
- 不做按小时段变化的节日商店刷新
- 不直接暴露底层 `month/day` 给大多数内容配置

## 记录策略

为了保证后续扩展清晰，每次季节/节日内容接入都应同步更新两处：

### 1. `docs/TODO.md`

明确记录：

- 本轮接入了哪些规则消费者
- 新增了哪些季节皮肤 / 节日内容
- 下一轮准备扩到哪些角色或哪些节日

### 2. 本规格文档

追加或更新：

- 当前已支持的皮肤可见性字段
- 当前已支持的内容条件字段
- 当前已有真实消费案例的标签

## 扩展方法

### 后续扩展到更多角色皮肤

下一步可按角色补：

- 默认季节皮肤
- 节日限定皮肤
- 誓约后的节日差分文本

### 后续扩展到活动商店

当 `festival_tags` 已稳定进入皮肤商店与内容层后，再考虑：

- 节日商品池
- 节日期间限定入口
- 活动期间特殊价格策略

### 后续扩展到天气系统

等天气系统落地后，可以让内容条件继续补：

- `required_weather`

但仍然应保持“先高层语义、后底层数据”的原则。

## 实现顺序建议

1. 扩展 `SkinDefinition` 的季节/节日可见性字段
2. 扩展 `SkinRuleEvaluator`
3. 扩展 `DialogueEntry` / `EventDefinition` 的时间语义条件
4. 扩展 `SceneContext` 的 `season` / `festival_tags`
5. 让 `skin_shop` 成为第一消费者
6. 给至少 1 位角色补季节/节日差分内容
7. 更新 `TODO` 与本规格文档中的已接入记录
