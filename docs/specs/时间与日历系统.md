# 时间与日历系统

## 概述

时间与日历系统负责 erAL 的真实日期、24 小时时钟、季节、节日、时间推进与日历查询能力。它是后续天气、活动商店、季节皮肤、节日口上和工作安排系统的共同底层。

本系统的核心约束是：

- 运行时主时间状态使用真实日期与 24 小时制
- 分钟是最小推进单位
- 移动与指令都必须消耗固定分钟数
- 旧 `time_slot` 机制只作为兼容派生层保留
- 日历指令当前只做纯查看，不提供预约或计划功能

## 当前阶段定位

- 当前阶段要先建立真实时间底座，而不是直接做活动商店或天气联动。
- 现有依赖 `time_slot` 的系统必须先兼容，不要求一次性全部重写。
- 日历指令当前只需要显示相邻几天的节日与舰娘工作安排。

## 数据结构

### CalendarState

世界当前日期状态。

| 字段 | 类型 | 说明 |
|------|------|------|
| `current_year` | int | 当前年份 |
| `current_month` | int | 当前月份 |
| `current_day` | int | 当前日期 |
| `current_weekday` | str | 当前星期，如 `mon` / `tue` |

### ClockState

世界当前时钟状态。

| 字段 | 类型 | 说明 |
|------|------|------|
| `current_hour` | int | 当前小时，`0-23` |
| `current_minute` | int | 当前分钟，`0-59` |

### Derived Time Slot

兼容旧系统的派生时段语义。它不是主状态，只由 `hour:minute` 计算得出。

建议映射：

| 时间范围 | 派生 `time_slot` |
|----------|------------------|
| `05:00-07:59` | `dawn` |
| `08:00-11:59` | `morning` |
| `12:00-16:59` | `afternoon` |
| `17:00-19:59` | `evening` |
| `20:00-23:59` | `night` |
| `00:00-04:59` | `late_night` |

### FestivalDefinition

节日与特殊日定义。

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 节日 key |
| `display_name` | str | 节日显示名 |
| `month` | int | 节日月份 |
| `day` | int | 节日日期 |
| `tags` | [str] | 节日标签，如 `summer_festival` / `new_year` |

### WorkScheduleDefinition

舰娘工作安排定义。它独立于普通位置日程。

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 排班 key |
| `actor_key` | str | 角色 key |
| `location_key` | str | 工作地点 |
| `work_key` | str | 工作类型 key |
| `work_label` | str | 工作显示名 |
| `start_time` | str | 开始时间，格式 `HH:MM` |
| `end_time` | str | 结束时间，格式 `HH:MM` |
| `date_rules` | dict | 生效日期规则 |

### CalendarDayView

提供给日历指令的结构化单日视图。

| 字段 | 类型 | 说明 |
|------|------|------|
| `year` | int | 年份 |
| `month` | int | 月份 |
| `day` | int | 日期 |
| `weekday` | str | 星期 |
| `season` | str | 当前季节 |
| `festival_labels` | [str] | 当天节日名 |
| `schedule_entries` | [WorkScheduleView] | 当天工作安排 |

### WorkScheduleView

日历中展示的一条工作安排。

| 字段 | 类型 | 说明 |
|------|------|------|
| `actor_name` | str | 角色名 |
| `time_range` | str | 例如 `09:00-12:00` |
| `location_name` | str | 地点名 |
| `work_label` | str | 工作内容 |

## 核心规则

### 真实时间是主状态

后续运行时唯一真实时间来源是：

`year / month / day / weekday / hour / minute`

不再使用：

`current_day + current_time_slot`

作为唯一时间真值。

### `time_slot` 只是兼容派生层

现有指令门禁、事件条件、旧日程表仍可继续读取 `time_slot`，但它必须由真实时钟计算得出，而不是单独维护。

### 所有行为必须声明耗时

任何能改变世界状态的行为，都必须显式声明分钟消耗。

当前阶段先使用固定分钟数，例如：

- 基础互动：`10 分钟`
- 移动：`15 分钟`
- 工作类指令：`30-60 分钟`
- 等待：`30 分钟`

后续若需要引入浮动耗时，应建立在固定耗时体系已经稳定的前提上。

### 工作安排独立于普通位置日程

现有 `schedule` 仍负责回答：

- 角色在这个时段在哪里

新增 `work_schedule` 负责回答：

- 角色在这个日期条件下、这个时间段里、在这个地点做什么工作

这两层不能继续混用，否则日历会重新退化成位置硬编码查询。

## 流程

### 时间推进

1. 行为声明固定耗时分钟数
2. 调用 `TimeService.advance(minutes)`
3. 处理分钟进位、小时进位、跨日、跨月、跨年
4. 重新计算：
   - `weekday`
   - `season`
   - `festival_tags`
   - 派生 `time_slot`
5. 触发依赖时间的刷新：
   - `ScheduleService`
   - 后续天气系统
   - 后续活动商店可见性

### 指令执行

1. 指令通过门禁校验
2. 指令执行主体逻辑
3. 指令按定义消耗固定分钟
4. 若时间跨越到新的派生时段，则日程与场景相关结果重新刷新
5. 结算、事件与口上继续读取新的世界时间状态

### 移动执行

1. 玩家选择移动
2. `NavigationService` 完成地点变化
3. 移动统一消耗固定分钟数，首版建议 `15 分钟`
4. 推进后刷新遭遇与时段语义

### 等待执行

1. 玩家执行 `wait`
2. 首版固定推进 `30 分钟`
3. 等待后刷新时间衍生语义和相关日程

### 日历指令

1. 玩家执行 `calendar`
2. `CalendarViewService` 以今天为中心构建固定窗口，首版建议 `前 2 天 + 今天 + 后 2 天`
3. 每一天显示：
   - 日期
   - 星期
   - 季节
   - 节日
   - 舰娘工作安排

## 配置

### calendar.toml

建议新增：`data/base/calendar.toml`

建议包含：

- 月份天数
- 星期显示名
- 季节映射
- 节日定义

示例：

```toml
[calendar]
weekday_names = ["mon", "tue", "wed", "thu", "fri", "sat", "sun"]
month_lengths = { 1 = 31, 2 = 28, 3 = 31, 4 = 30, 5 = 31, 6 = 30, 7 = 31, 8 = 31, 9 = 30, 10 = 31, 11 = 30, 12 = 31 }

[[seasons]]
key = "spring"
months = [3, 4, 5]

[[seasons]]
key = "summer"
months = [6, 7, 8]

[[festivals]]
key = "summer_festival"
display_name = "夏日祭"
month = 6
day = 21
tags = ["summer", "festival"]
```

### work_schedules.toml

建议新增：`data/base/work_schedules.toml`

示例：

```toml
[[work_schedules]]
key = "enterprise_office_weekday_morning"
actor_key = "enterprise"
location_key = "command_office"
work_key = "office_duty"
work_label = "文书值班"
start_time = "09:00"
end_time = "12:00"

[work_schedules.date_rules]
weekdays = ["mon", "tue", "wed", "thu", "fri"]

[[work_schedules]]
key = "laffey_dock_summer_patrol"
actor_key = "laffey"
location_key = "dock"
work_key = "summer_patrol"
work_label = "夏季巡逻"
start_time = "14:00"
end_time = "18:00"

[work_schedules.date_rules]
months = [6, 7, 8]
festival_tags = ["summer_festival"]
```

## 与现有系统的关系

- 依赖 [日程系统](日程系统.md)
  - 普通角色位置刷新暂时继续读派生 `time_slot`
- 依赖 [场景上下文系统](场景上下文系统.md)
  - 场景上下文后续需要补真实日期和时钟字段
- 依赖 [事件系统](事件系统.md)
  - 节日与具体时间将成为新条件来源
- 依赖 [口上系统](口上系统.md)
  - 节日、季节与精确时间可成为口上筛选条件
- 依赖 [商店与经济系统](商店与经济系统.md)
  - 节日与季节规则最终会驱动活动商店或皮肤上架
- 依赖 [皮肤与外观系统](皮肤与外观系统.md)
  - `summer`、`festival` 等标签最终从时间系统取语义来源

## 兼容策略

### 第一阶段

- 先把底层主时间切到真实日期 + 时钟
- 保留 `time_slot` 派生接口
- 旧内容继续按 `time_slot` 运作

### 第二阶段

- 把 `SceneContext` 扩展为：
  - `year`
  - `month`
  - `day`
  - `weekday`
  - `hour`
  - `minute`
  - `season`
  - `festival_tags`
  - 兼容的 `time_slot`

### 第三阶段

- 逐步把旧 `time_slots` 和 `schedule` 内容迁移到更细粒度的时间规则
- 让活动商店、季节皮肤、节日口上直接消费真实日期语义

## 当前阶段不做

- 不做玩家预约或日历计划系统
- 不做天气联动
- 不做活动商店自动上架
- 不做任意月历翻页
- 不做复杂排班冲突处理
- 不做可变耗时或距离耗时

## 扩展方法

### 加天气系统

天气系统后续应读取：

- 当前日期
- 当前时段
- 当前季节

而不是自建平行时间状态。

### 加节日活动商店

后续商店系统可直接读取 `festival_tags` 或 `season` 判定活动商品是否可见。

### 加皮肤活动上架

后续 `SkinRuleEvaluator` 可直接消费：

- `season`
- `festival_tags`
- 真实日期范围

### 加更细时间条件

后续事件、口上和工作安排可继续扩展：

- `time_range`
- `weekdays`
- `festival_tags`

但前提是统一从时间系统读取，而不是各自硬编码判断。

## 实现顺序建议

1. 扩展 `WorldState` 为真实日期 + 24 小时时钟
2. 新增 `TimeService`
3. 保留 `time_slot` 派生兼容层
4. 让 `wait` 和 `NavigationService` 按分钟推进
5. 给 `CommandDefinition` 增加固定耗时字段
6. 让 `CommandService` 统一推进时间
7. 新增 `calendar.toml`
8. 新增 `work_schedules.toml`
9. 新增 `CalendarService` 与 `CalendarViewService`
10. 实现日历指令 CLI 入口
11. 最后再让商店、皮肤、事件逐步接入季节与节日规则
