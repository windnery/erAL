# 事件系统

## 概述

事件系统根据场景上下文匹配静态事件定义，产出事件 key 列表供口上系统使用。事件是口上选择的第一优先级——先按事件 key 查找口上，无匹配则回退到指令 key。

## 数据结构

### EventDefinition

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 唯一标识 |
| `action_key` | str | 触发指令 key（与指令 key 匹配） |
| `actor_tags` | [str] | 角色标签过滤（OR 匹配） |
| `location_keys` | [str] | 地点过滤（OR 匹配） |
| `time_slots` | [str] | 时段过滤 |
| `min_affection` | int? | 最低好感 |
| `min_trust` | int? | 最低信赖 |
| `min_obedience` | int? | 最低服从 |
| `required_stage` | str? | 最低关系阶段 |
| `requires_date` | bool? | 是否需要约会 |
| `requires_private` | bool | 是否需要私密场景 |
| `required_marks` | {str: int} | 需要的标记 |

## 流程

```
1. CommandService 构建 trigger_scene（结算后、操作前）
2. EventService.triggered_events(scene) 遍历所有事件定义：
   - action_key 不匹配 → 跳过
   - actor_tags 无交集 → 跳过
   - location_keys 不包含当前地点 → 跳过
   - time_slots 不包含当前时段 → 跳过
   - min_affection/trust/obedience 不足 → 跳过
   - required_stage rank 不够 → 跳过
   - requires_date 不满足 → 跳过
   - requires_private 但场景非私密 → 跳过
   - required_marks 任一不满足 → 跳过
3. 全部条件通过 → 加入 matched 列表
4. 返回 matched 事件 key 元组
```

**当前行为**：所有条件满足的事件都会触发（全匹配），无加权随机，无互斥选择。未来事件数量增长后需引入权重和互斥机制。

## 配置

事件定义分两层：

- **全局事件**：`data/base/events.toml`（目前为空或不存在则跳过）
- **角色事件**：`data/base/characters/<key>/events.toml`

角色事件通过 `actor_tags` 过滤，同一 action_key 可匹配多个角色的事件。

## 扩展方法

1. **加新事件**：在角色包的 `events.toml` 中加 `[[events]]`，指定 action_key 和条件
2. **加新条件维度**：在 EventDefinition 增加字段，在 EventService.triggered_events() 增加过滤逻辑
3. **引入权重**：未来在 EventDefinition 加 weight 字段，匹配后按权重随机选择而非全触发
