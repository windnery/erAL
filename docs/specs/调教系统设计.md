# 调教系统设计

## 概述

调教系统是 erAL 后续的核心玩法系统之一。

它负责：

- 提供区别于日常 / 约会 / 工作的独立玩法模式
- 管理调教过程中的指令判定、执行与结算
- 推进长期调教相关数值
- 为事件、口上、存档与后续角色内容提供统一语义

本系统的目标不是照搬 erArk 的复杂身体管理、器具和捆绑体系，而是：

- 至少达到 eratw 级别的调教系统规模
- 保持与当前 `commands + gates + scene + settlement + marks + save/load` 架构兼容
- 先做可玩的完整闭环，再扩到长期训练与内容消费
- 明确避开“器具系统化 / 捆绑系统化”这条扩展路线

## 设计目标

### 1. 第一版要达到“eratw 规模”，但不一次性做成重系统

第一版调教系统不是只放几条调情指令的轻量补丁，而是要成为一个独立的核心玩法层。

但它也不应一开始就做成：

- erArk 那种复杂器具管理
- 捆绑姿势层级系统
- 细粒度身体部位容器化

正确方向是：

- 先做一个完整可玩的调教系统
- 再在第二阶段补长期推进和内容消费

### 2. 优先复用现有架构，不重造平行系统

当前项目已经有：

- `commands.toml`
- `CommandService`
- `CommandAvailabilityGate`
- `SceneService`
- `SettlementService`
- `marks`
- `removed_slots`
- `save/load`

调教系统应尽量建立在这些底座上，而不是再新造一套与主命令系统平行的执行器。

### 3. 服装脱除继续沿用现有皮肤/外观运行时层

用户已经明确要求：

- 皮肤是整套切换，不允许自由换装
- 调教过程中衣服可以被逐步脱除
- 脱除后会影响可执行指令

因此调教系统必须直接消费现有的：

- `equipped_skin_key`
- `removed_slots`

而不是引入新的装备状态模型。

## 明确不做

第一版和当前规划中，明确不做：

- 器具装备系统化
- 捆绑类型与姿势系统化
- 复杂 body part 数据模型
- 液体流向 / 身体容器模拟
- 高成本连续 AI 决策
- Web 交互式调教 UI

如果后续要扩展，也应优先沿着：

- 指令族扩展
- 长期训练数值扩展
- 内容条件扩展

而不是先走器具 / 捆绑方向。

## 两段式里程碑

调教系统整体按两个里程碑推进。

### 里程碑 A：可玩版

目标：

- 调教成为一个可进入、可执行、可退出的完整玩法模式
- 有主干指令族
- 有门禁
- 有衣物脱除
- 有基础数值推进

包含：

- 调教模式状态
- 调教专用门禁
- 轻爱抚 / 手 / 口 / 基础插入等主干指令族
- 运行时衣物脱除
- 基础调教数值写入

### 里程碑 B：完整版

目标：

- 达到“至少 eratw 规模”的完整调教系统
- 具备长期推进与内容消费能力

包含：

- 高潮 / 拒绝 / 中断 / 失败状态
- 长期开发推进
- 调教结果与事件 / 口上联动
- 更完整的存档兼容和回归测试

## 第一版总体架构

### 1. 调教模式状态层

建议在 `WorldState` 中新增一个轻量调教会话状态，例如：

- 是否处于调教模式
- 当前调教对象
- 当前调教场景
- 当前体位或调教阶段
- 当前会话内的短期状态

该状态的职责是：

- 让系统知道当前不是普通互动
- 让后续调教指令共享同一个会话上下文
- 为中断、退出、存档恢复提供依据

这层只保存“当前会话”状态，不直接承载长期数值。

### 2. 调教命令层

调教仍走 `CommandService` 主命令管线，但会新增一组调教专用命令。

这些命令应继续复用：

- `CommandDefinition`
- `required_stage`
- `required_conditions`
- `required_marks`
- `location_tags`
- `time_slots`

但需要增加调教专用语义，例如：

- 是否要求处于调教模式
- 是否要求已脱除某些服装槽位
- 是否要求某类调教前置已经发生
- 是否要求某些长期调教值达到阈值

### 3. 调教门禁层

调教门禁是本系统的关键。

除了现有门禁外，建议增加一层调教专用 gate，统一处理：

- 当前是否已进入调教
- 当前角色是否允许进入调教
- 当前服装脱除条件是否满足
- 当前体位 / 会话阶段是否允许该指令
- 当前调教数值是否达到前置要求

这样可以避免把调教规则散写在各条命令实现里。

### 4. 调教结算层

调教结算继续复用 `SettlementService` 的思路，但应有独立调教结算语义。

核心原则：

- 日常命令的 settlement 不应和调教命令完全混在一起
- 调教命令需要一套更偏“快感 / 羞耻 / 顺从 / 开发”的结算逻辑

建议做法：

- 仍然通过 `source -> settlement` 管线推进
- 但在 settlement 规则中明确区分调教类 source
- 调教类结算负责写入调教相关 PALAM / 长期值 / mark

## 核心数据模型建议

### 1. 会话级状态

建议第一版至少有：

- `train_actor_key`
- `train_mode_active`
- `train_position_key`
- `train_step_index`
- `train_session_flags`

用途：

- 记录当前调教对象
- 记录当前会话上下文
- 支持中断、退出、继续执行

### 2. 调教核心数值

第一版建议分成两类。

#### A. 短中期过程值

优先使用现有 PALAM 扩展或复用：

- `pleasure_c`
- `pleasure_v`
- `pleasure_a`
- `pleasure_b`
- `pleasure_m`
- `submission`
- `shame`
- `fear`
- `lust`
- `obedience`

这些值用于：

- 指令即时结算
- 高潮 / 拒绝 / 顺从推进判断
- 口上和事件条件

#### B. 长期推进值

建议放在里程碑 B 再完整接入：

- C 开发
- V 开发
- A 开发
- 奉仕系开发
- 插入系经验或长期训练值

这些值的作用是：

- 解锁更深层指令
- 改变角色长期反应
- 为后续事件和分支提供条件

### 3. 衣物状态

衣物不新建模型，直接复用：

- `removed_slots`

建议调教命令直接以服装槽位为前置条件，例如：

- 基础插入前必须脱除 `underwear_bottom`
- 某些胸部相关指令必须脱除 `top` 或胸部遮挡位

## 指令族拆分建议

第一版不要按零散单指令推进，而要按“族”推进。

### 里程碑 A 应覆盖的指令族

#### 1. 轻爱抚系

- 抚摸
- 胸部爱抚
- 下体爱抚
- 调情 / 挑逗 / 接吻延伸

作用：

- 建立最轻量的调教入口
- 主要推进快感、羞耻、情欲等值

#### 2. 手系

- 手交
- 胸交前置

作用：

- 让调教从轻接触进入明确 H 指令

#### 3. 口系

- 口交
- 奉仕基础分支

作用：

- 形成主干指令树的第二层

#### 4. 基础插入系

- V 插入
- A 插入

作用：

- 形成真正的调教主干
- 同时验证衣物脱除与门禁链路

#### 5. 管理系

- 脱除上衣
- 脱除内裤
- 切换体位
- 结束调教
- 中断调教

作用：

- 提供调教流程管理能力

### 里程碑 B 再扩展的内容

- 更完整的体位差分
- 更完整的插入派生指令
- 更复杂的奉仕 / 屈服路线
- 开发度驱动的解锁

## 结果状态与过程反馈

这是调教系统能否接近 eratw 规模的关键。

### 1. 基础结果态

第一版建议至少支持：

- 成功执行
- 拒绝
- 中断
- 高潮 / 绝顶

### 2. 高潮

高潮不应只是文本，而应是明确结算状态。

它应至少影响：

- 当前会话推进
- 后续指令效果
- 对应口上与事件命中
- 某些长期值推进

### 3. 拒绝 / 中断

拒绝和中断应区分：

- 拒绝：门禁或执行中判断失败
- 中断：体力、理性、状态或会话原因导致流程被迫停止

### 4. 长期推进

当里程碑 B 接入后，调教不再只是短期快感推进，而会改变角色长期状态：

- 指令可用性
- 反应模式
- 内容分支

## 与现有系统的关系

### 1. 与皮肤 / 外观系统

直接复用：

- `removed_slots`

调教过程中脱衣是“运行时覆盖”，不是换装。

### 2. 与场景上下文系统

调教必须扩展 `SceneContext`，至少让以下信息可被消费：

- 当前是否处于调教模式
- 当前体位
- 当前会话阶段
- 当前脱除槽位
- 当前调教结果态

### 3. 与事件 / 口上系统

调教系统后续必须提供可消费条件，例如：

- 调教中
- 已高潮
- 已拒绝
- 已进入插入阶段
- 已脱除某些槽位

### 4. 与存档系统

调教会话状态和长期调教值必须可存档。

同时必须考虑：

- 旧档缺字段的默认降级
- 中途处于调教中的存档如何处理

第一版可接受的策略是：

- 旧档默认不在调教中
- 缺少调教字段时回退到安全默认值

## 测试策略

调教系统的测试必须分层。

### 1. 门禁测试

覆盖：

- 关系不足
- 衣物未脱除
- 不在调教中
- 会话阶段不符
- 长期值不足

### 2. 执行测试

覆盖：

- 轻爱抚类
- 手系
- 口系
- 基础插入系

### 3. 结果测试

覆盖：

- 高潮
- 拒绝
- 中断
- 正常结束

### 4. 存档测试

覆盖：

- 调教字段 roundtrip
- 旧档缺调教字段降级

## 推荐实施顺序

### 里程碑 A：可玩版

1. 调教会话状态
2. 调教门禁层
3. 调教最小指令族
4. 衣物脱除与插入前置
5. 基础调教结算

### 里程碑 B：完整版

1. 高潮 / 拒绝 / 中断机制
2. 长期开发度
3. 调教结果条件接入事件 / 口上
4. 更完整回归与平衡

## 工程建议

不要一次性把“eratw 规模调教系统”做成一个单阶段任务。

正确拆法应该是：

- 里程碑 A：可玩版
- 里程碑 B：完整版

这样既能尽快形成可玩的系统闭环，也不会一开始就把复杂度拉到不可控。

## 结论

调教系统推荐采用：

- 目标规模参考 `eratw`
- 工程架构复用当前 `erAL`
- 扩展方向避开 `erArk` 的器具 / 捆绑路线
- 实施阶段拆成“可玩版 + 完整版”两段

这是当前项目下最稳、最符合你长期方向的实现路线。
