# 系统总览与缺口（Spec 视角）

更新时间：2026-04-22

目的：从规格层说明“哪些系统已具备工程骨架，哪些仍需要补消费者与反馈链路”。

## 1. 现状判断

结论：erAL 不是系统缺失型项目，而是系统表达不均衡型项目。

当前主问题：

1. 主链完整，但部分数值轴反馈弱。
2. 子系统数量充足，但高复杂模块的自动化验证仍需加强。
3. 内容规模已过原型线，文档体系需要持续对齐真实实现。

## 2. 已具备稳定骨架的系统

以下系统已能稳定支撑成品化迭代：

1. 指令主链：commands / command_gates / settlement。
2. 场景反馈：scene / events / dialogue。
3. 关系推进：relationships + stages。
4. 回合与世界：game_loop / time / schedule / navigation / distribution。
5. 经济资产：wallet / shop / facilities / commissions / skins / gifts。
6. 持久化：save + runtime 状态目录。

## 3. 需要优先补强的缺口

### 3.1 数值轴消费者缺口

症状：部分 TALENT/PALAM 仍存在“有数据、弱消费”。

影响：玩家不容易感知数值变化，系统深度被折损。

补强方向：

1. 先补高频行为会触发的消费者。
2. 每补一条规则必须附带至少一个测试用例。
3. 规则文档必须说明“由谁读取、影响什么反馈”。

### 3.2 回归保障缺口

症状：个别复杂系统（如训练相关链路）对内容扩张的抗回归能力不足。

影响：新增内容时可能引入隐性行为回退。

补强方向：

1. 建立主链金路径回归。
2. 对高风险模块增加最小可复现回归样例。
3. 内容校验增加坏样例夹具。

### 3.3 表达一致性缺口

症状：系统行为存在，但玩家反馈强度不一致。

影响：玩家无法稳定感知系统变化。

补强方向：

1. 统一指令失败反馈模板。
2. 强化状态变化提示（关系、体力、关键标记）。
3. 事件/口上 fallback 规则持续补齐。

## 4. 规格层实施优先级

1. P0：TALENT/PALAM 关键消费者补齐。
2. P0：主链回归测试稳定化。
3. P1：高频指令反馈一致性。
4. P1：内容校验工具增强。
5. P2：扩展系统深度与内容工业化配套。

## 5. 本文档的使用方式

1. 立项前先对照本文件确认是否命中优先缺口。
2. 实施中将改动映射到具体 spec 文档。
3. 收尾时回填“缺口是否被消除”并更新优先级。
