# 皮肤与外观系统

## 概述

皮肤与外观系统负责角色整套外观的拥有、获取、装备与内容语义分支。它的核心约束是：

- 玩家只能选择切换`皮肤`
- 皮肤切换后，整套服装槽位自动切到该皮肤预设
- 不支持自由搭配服装
- 调教过程中允许系统临时脱除部分服装槽位，但这不是自由换装
- 皮肤允许携带非数值语义标签
- 皮肤不提供数值效果，不参与结算公式

本系统的目标不是做一个“装备系统”，而是做一个“角色外观方案系统”。对舰娘来说，皮肤是一整套预设，服装槽位只是这套预设展开后的结构化结果。

## 当前阶段定位

- 当前阶段先支持 `skin_shop`、誓约奖励皮肤、当前皮肤切换与状态显示。
- 立绘资源切换接口会预留，但立绘渲染本身可以后置。
- 季节/日期限制会预留规则接口，但当前不依赖完整季节系统。

## 数据结构

### SkinDefinition

玩家真正拥有、购买、切换的对象。

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 皮肤 key |
| `actor_key` | str | 所属角色 key |
| `display_name` | str | 皮肤显示名 |
| `price` | int | 皮肤价格 |
| `grant_mode` | str | 获取方式：`shop` / `oath_reward` |
| `shop_visibility` | str | 上架规则：`always` / `manual` |
| `tags` | [str] | 语义标签，如 `summer` / `swimsuit` / `oath` |
| `appearance_key` | str | 对应的外观方案 key |

### AppearanceDefinition

皮肤展开后的整套外观结果。

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 外观方案 key |
| `portrait_key` | str | 立绘资源 key，当前阶段可只做预留 |
| `slots` | dict[str, str] | 服装槽位到资源 key 的映射 |

### Outfit Slots

当前版本固定为 9 个槽位，不开放自由拼装：

| 槽位 key | 中文名 | 说明 |
|----------|--------|------|
| `headwear` | 头饰 | 帽子、发箍、头冠等 |
| `accessory` | 饰品 | 项链、手环、胸针等 |
| `top` | 上衣 | 主上装 |
| `skirt` | 裙子 | 下装，当前统一用裙子槽位承载 |
| `underwear_top` | 内衣 | 内层上装 |
| `underwear_bottom` | 内裤 | 内层下装 |
| `socks` | 袜子 | 长袜、短袜、裤袜等 |
| `shoes` | 鞋子 | 鞋履 |
| `attachment` | 附属 | 披肩、尾饰、背饰、特殊挂件等 |

### AppearanceState

角色运行时外观状态。它应挂在角色身上，而不是全局背包里。

| 字段 | 类型 | 说明 |
|------|------|------|
| `owned_skins` | [str] | 已拥有皮肤 key 列表 |
| `equipped_skin_key` | str? | 当前装备皮肤 key |

### ExposureState

调教、特殊事件或剧情过程中的临时脱衣状态。它不是玩家可编辑的装备状态，而是覆盖在当前皮肤展开结果上的运行时层。

| 字段 | 类型 | 说明 |
|------|------|------|
| `removed_slots` | [str] | 当前被临时脱除的槽位 key |
| `reason` | str? | 状态来源，如 `train_session` / `event` |

### Scene Appearance Context

为了让口上和事件能按皮肤分支，场景上下文后续需要补充：

| 字段 | 类型 | 说明 |
|------|------|------|
| `equipped_skin_key` | str? | 当前装备皮肤 |
| `equipped_skin_tags` | [str] | 当前皮肤标签 |
| `removed_slots` | [str] | 当前被临时脱除的服装槽位 |

## 核心规则

### 单一状态源

运行时只允许存在一个玩家可操作的外观状态源：`equipped_skin_key`。

服装槽位不单独作为可编辑状态保存，而是通过如下链路自动展开：

`equipped_skin_key -> appearance_key -> slots`

调教或剧情中的脱衣效果则作为临时运行时覆盖层存在：

`equipped_skin_key -> appearance_key -> slots -> removed_slots override`

这条规则的目的是避免以下冲突：

- 玩家装备了 A 皮肤，但服装槽位仍残留 B 皮肤数据
- UI 切了皮肤，但事件系统读取到的是旧服装状态
- 立绘系统和槽位系统分别维护两套不同的当前外观
- 调教过程中脱掉了内裤，却被错误保存成“永久换装”

### 脱衣不是换装

后续调教系统上线后，允许它在运行时临时移除部分服装槽位，例如：

- 脱掉 `underwear_bottom` 后，插入系指令才允许执行
- 脱掉 `top` / `skirt` 后，命中特定暴露或羞耻分支

但这些变化必须满足：

- 只能由系统流程驱动，不提供玩家自由编辑界面
- 只影响当前过程中的可见/可用槽位
- 不改变角色的 `equipped_skin_key`
- 流程结束、离开场景或状态重置后，应恢复到当前皮肤的默认展开结果

也就是说：

- `皮肤切换` = 长期外观状态
- `脱衣状态` = 短期运行时覆盖

### 皮肤不是数量型道具

皮肤不能落在普通 `inventory` 中。

原因：

- 普通 `inventory` 语义是“数量型、可消费、可堆叠物品”
- 皮肤语义是“角色绑定、永久解锁、可装备内容”

因此皮肤购买成功后的落点是角色的 `owned_skins`，不是全局背包。

### 皮肤不提供数值效果

皮肤不会：

- 提高属性
- 改变指令成功率
- 影响 SOURCE / PALAM / CFLAG 结算

皮肤只会影响：

- 是否上架
- 是否可获取
- 是否可装备
- 当前外观显示
- 内容分支匹配
- 某些指令的暴露前提判定

## 流程

### 皮肤购买

1. 玩家进入 `skin_shop`
2. 系统读取该店面可见的 `SkinDefinition`
3. `SkinRuleEvaluator` 判断皮肤当前是否允许上架
4. 玩家购买后扣款
5. 将皮肤 key 写入目标角色的 `owned_skins`
6. 不写入 `inventory`

### 皮肤装备

1. 玩家在角色外观界面选择某个已拥有皮肤
2. 系统校验该皮肤属于该角色，且当前允许装备
3. 写入 `equipped_skin_key`
4. UI 与后续内容系统统一从该 key 展开外观方案

### 调教中的临时脱衣

1. 调教系统进入某个阶段
2. 系统按当前行为移除指定槽位，例如 `underwear_bottom`
3. 写入 `ExposureState.removed_slots`
4. 指令门禁、事件和口上读取当前 `removed_slots`
5. 当流程结束或重置时，清空 `removed_slots`
6. 角色外观恢复为当前皮肤的默认 9 槽位结果

### 誓约奖励皮肤

1. `oath` 成功
2. 系统检查该角色是否存在 `grant_mode = "oath_reward"` 的皮肤
3. 若存在且尚未拥有，则自动写入 `owned_skins`
4. 默认不强制自动装备，由玩家自行切换

### 内容分支

1. 场景构建时读取当前角色的 `equipped_skin_key`
2. 将 `equipped_skin_key` 与 `equipped_skin_tags` 写入 `SceneContext`
3. 事件和口上可额外声明：
   - `required_skin_key`
   - `required_skin_tags`
4. 当这些条件满足时，命中特定皮肤分支文本

后续若需要支持脱衣分支，可增加：

- `required_removed_slots`
- `forbidden_removed_slots`

## 规则解释层

建议新增 `SkinRuleEvaluator`，统一解释皮肤资格规则，避免把判断散在 UI、商店、誓约和内容系统里。

当前阶段先支持以下最小规则：

| 字段 | 值 | 当前含义 |
|------|----|---------|
| `grant_mode` | `shop` | 可从 `skin_shop` 购买 |
| `grant_mode` | `oath_reward` | 誓约成功后自动获得 |
| `shop_visibility` | `always` | 常驻上架 |
| `shop_visibility` | `manual` | 先保留数据，不默认上架 |

标签仅作为语义信息存在，例如：

- `summer`
- `swimsuit`
- `oath`
- `casual`
- `festival`

当前阶段这些标签不直接驱动复杂时间判定，只用于：

- 商店分类
- 内容筛选
- 后续系统扩展

## 配置

### skins.toml

建议新增：`data/base/skins.toml`

```toml
[[skins]]
key = "enterprise_default"
actor_key = "enterprise"
display_name = "默认服装"
price = 0
grant_mode = "shop"
shop_visibility = "manual"
tags = ["default"]
appearance_key = "enterprise_default"

[[skins]]
key = "enterprise_oath"
actor_key = "enterprise"
display_name = "誓约礼服"
price = 0
grant_mode = "oath_reward"
shop_visibility = "manual"
tags = ["oath", "formal"]
appearance_key = "enterprise_oath"

[[skins]]
key = "enterprise_summer"
actor_key = "enterprise"
display_name = "夏日泳装"
price = 1680
grant_mode = "shop"
shop_visibility = "manual"
tags = ["summer", "swimsuit"]
appearance_key = "enterprise_summer"
```

### appearances.toml

建议新增：`data/base/appearances.toml`

```toml
[[appearances]]
key = "enterprise_oath"
portrait_key = "enterprise_oath_main"

[appearances.slots]
headwear = "oath_crown"
accessory = "oath_ring"
top = "oath_dress_top"
skirt = "oath_dress_skirt"
underwear_top = "oath_inner_top"
underwear_bottom = "oath_inner_bottom"
socks = "oath_stockings"
shoes = "oath_heels"
attachment = "oath_veil"
```

### 角色状态存档

角色存档需增加：

- `owned_skins`
- `equipped_skin_key`

旧存档兼容策略：

- `owned_skins` 缺失时，默认回填角色默认皮肤
- `equipped_skin_key` 缺失时，默认装备默认皮肤

## 与现有系统的关系

- 依赖 [商店与经济系统](商店与经济系统.md)
  - `skin_shop` 复用商店入口与扣款逻辑
- 依赖 [口上系统](口上系统.md)
  - 皮肤条件会扩展 DialogueEntry 的条件匹配
- 依赖 [事件系统](事件系统.md)
  - 某些活动或誓约后事件可按皮肤分支
- 依赖 [关系扩展系统](关系扩展系统.md)
  - `oath_reward` 皮肤由誓约成功发放
- 依赖 [场景上下文系统](场景上下文系统.md)
  - 当前皮肤状态需要进入 SceneContext

## 当前阶段不做

- 不做自由搭配服装
- 不做数值型皮肤效果
- 不做完整季节/日期上架判定
- 不做立绘资源实际渲染
- 不做身体部位级拆分换装
- 当前阶段不实现调教过程中的脱衣结算，只预留状态结构与接口

## 扩展方法

### 加活动或季节皮肤

后续只扩展 `SkinRuleEvaluator`，不改状态结构。

未来可新增字段：

- `shop_visibility = "seasonal"`
- `season = "summer"`
- `date_window`
- `event_key`

### 加皮肤专属口上

在事件或口上条件中新增：

- `required_skin_key`
- `required_skin_tags`

从而支持：

- 誓约皮肤专属晨间口上
- 泳装皮肤专属夏季互动
- 节日皮肤专属活动文本

### 加调教中的暴露判定

调教系统后续可直接读取 `removed_slots`，而不是修改皮肤或服装定义本身。

例如：

- 插入系指令要求 `underwear_bottom` 已被移除
- 某些羞耻事件要求 `top` 或 `skirt` 已被移除

这样可以保证：

- 脱衣条件可被指令门禁复用
- 外观恢复逻辑统一
- 不会把临时脱衣误写进长期皮肤状态

### 加立绘系统

立绘系统应直接读取：

- `equipped_skin_key`
- `appearance_key`
- `portrait_key`

而不是建立第二套外观状态。

## 实现顺序建议

1. 新增 `skins.toml` 与 `appearances.toml`
2. 新增加载器与定义对象
3. 扩展 `CharacterState` 和存档结构
4. 新增 `SkinRuleEvaluator`
5. 让 `skin_shop` 支持列货与购买
6. 支持角色皮肤切换
7. 接入誓约奖励皮肤自动发放
8. 将皮肤状态接入口上和事件条件
9. 为调教系统预留 `removed_slots` 判定接口
10. 最后接入立绘显示
