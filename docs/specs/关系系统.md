# 关系系统

## 概述

关系系统根据好感（affection）、信赖（trust）和亲密值（intimacy）计算角色当前的关系阶段。阶段影响指令可用性、事件触发和口上选择。阶段不是直接设置的，而是由数值和状态条件推算得出。

自 2026-04-12 起，运行时代码通过 `src/eral/domain/compat_semantics.py` 提供的英文 key 访问这些 compat 轴；玩家可见语义仍以 era 风格标签为准。

## 数据结构

### RelationshipStage（运行时阶段）

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 阶段标识：stranger/friendly/like/love/oath |
| `display_name` | str | 显示名称 |
| `rank` | int | 阶段序号（0 起），用于比较高低 |

### RelationshipStageDefinition（静态定义）

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 阶段标识 |
| `display_name` | str | 显示名称 |
| `min_affection` | int | 最低好感 |
| `min_trust` | int | 最低信赖 |
| `min_intimacy` | int | 最低亲密值，当前由 ABL:9 同步 |
| `no_dislike_mark` | bool | 是否禁止带 dislike_mark 进入该阶段 |
| `requires_item` | str | 预留的道具门槛；当前配置可写，运行时暂未启用 |

## 流程

### 阶段推算

```
1. 遍历所有阶段定义（按 min_affection + min_trust 升序排列）
2. 如果当前 affection ≥ stage.min_affection 且 trust ≥ stage.min_trust → 记录为候选
3. 亲密值必须同时满足 stage.min_intimacy
4. 若 stage.no_dislike_mark 为真，且角色持有 dislike_mark，则跳过
5. 最终选最后一个满足条件的阶段（最高匹配）
6. 如果没有任何阶段满足 → stranger（rank=0）
```

**语义**：阶段只升不降——即使好感/信赖因特殊原因降低，当前代码不会回退阶段。未来可能需要处理阶段回退。

### 阶段更新时机

```
1. SettlementService.settle_actor() 结算后自动调用 RelationshipService.update_actor()
2. `CharacterState.sync_derived_fields()` 通过 compat 语义访问层同步 `affection / trust / obedience`
3. 测试中手动调用 RelationshipService.update_actor(actor)
```

### 阶段在条件匹配中的使用

- **指令**：`required_stage` → 比较 rank
- **事件**：`required_stage` → 比较 rank
- **口上**：`required_stage` → 精确匹配 stage key（不是 rank 比较）

> `requires_item` 目前只在配置层保留，还没有接入运行时判定。后续若要启用誓约道具或阶段钥匙，建议优先放进指令门控层统一处理。

> **注意**：指令和事件用 rank 比较（≥），口上用 key 精确匹配（==）。这是历史遗留的不一致，未来应统一。

## 配置

文件路径：`data/base/relationship_stages.toml`

| 阶段 | display_name | min_affection | min_trust |
|------|-------------|---------------|-----------|
| stranger | 陌生 | 0 | 0 |
| friendly | 友好 | 1 | 0 |
| like | 喜欢 | 3 | 2 |
| love | 爱 | 6 | 4 |
| oath | 誓约 | 10 | 6 |

## 扩展方法

1. **加新阶段**：在 `relationship_stages.toml` 中加 `[[stages]]`，注意 min_affection/min_trust 必须递增
2. **调整阶段门槛**：修改对应阶段的 min_affection/min_trust/min_intimacy
3. **加阶段特效**：目前阶段仅影响条件匹配，未来可在阶段变化时触发事件或口上
4. **加道具门槛**：如果后面真的要用 `requires_item`，先把它接进门控层，再改动阶段解析逻辑
