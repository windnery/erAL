# 地图分层与分布系统

## 概述

地图分层与分布系统负责：

- 定义港区的空间层级结构
- 决定玩家可到达的具体地点
- 管理大量舰娘在地图上的分布与分流
- 为后续商店、委托、天气、调教、活动和地图 UI 提供统一空间语义

本系统的设计目标不是做一个复杂 AI，而是做一个：

- 细颗粒地点可扩展
- 高角色量可承载
- 规则驱动、低性能负担
- 方便后续整改与扩容

的港区空间骨架。

## 设计原则

### 1. 顶层采用“公共功能区 + 阵营生活区”的混合结构

地图最高层不完全按阵营，也不完全按功能切分，而是两种并存：

- 公共功能区
  - 指挥中枢区
  - 训练演习区
  - 商业休闲区
  - 港口外勤区
  - 访客/活动区

- 阵营生活区
  - 皇家生活区
  - 白鹰生活区
  - 重樱生活区
  - 铁血生活区
  - 东煌生活区
  - 混合生活区

这样既能体现碧蓝航线的阵营特色，也能避免把后勤、训练、商店等公共功能重复铺到每个阵营里。

### 2. 地图保持细颗粒地点，但部分高频玩法区域允许“功能合并”

整体地图依然采用细颗粒地点，不做“整个大区就是一个地点”的粗化设计。

但某些区域可以把多个玩法入口合并到一个关键细地点里，以减少玩家来回跑图。

典型案例：

- `指挥中枢区 -> 指挥室`
  - 玩家到达 `指挥室` 后即可执行：
    - 工作类指令
    - 委托类指令
    - 后续出击准备类指令

这意味着地图依然细分，但办公相关玩法入口可以集中到一个核心地点。

### 3. 商店作为共享地点，后续由 NPC 服务语义分流

商业休闲区内当前只需要一个 `商店` 地点。

后续 `明石` 与 `不知火` 可以作为常驻舰娘在该地点出现，并通过统一的 `购物` 指令分流到底层不同店面：

- 明石 -> `skin_shop`
- 不知火 -> `general_shop`

也就是说：

- 地图层是一个商店
- 服务语义层是多个 shopfront
- NPC 层负责入口风味和角色表现

### 4. 委托中的舰娘应从地图上移除

委托不仅是玩法状态，也是地图减压机制的一部分。

当舰娘处于：

- 委托中

时，应当：

- 不出现在地图地点分布里
- 不占用任何地点承载位
- 不参与地点内舰娘列表

这样可以天然减轻高角色量下的地图拥挤问题。

### 5. 非工作状态舰娘采用“轻量规则驱动动态分布”，不做复杂 AI

后续非工作状态舰娘不应完全站桩，也不应采用高成本的持续 AI 模拟。

推荐做法：

- 候选地点集合
- 静态权重
- 时段修正
- 饭点修正
- 阵营修正
- 个性修正
- 与玩家关系修正
- 地点承载分流

最后通过规则选出当前地点。

这是一套：

- 低频刷新
- 权重驱动
- 非 AI
- 易调试

的动态分布系统。

### 6. 地点内舰娘全部可见，但 UI 采用分页与自动选中

玩家进入一个具体地点后：

- 该地点内的舰娘理论上全部可见
- UI 可按分页显示，建议每页最多 `10` 人
- 只展示当前页列表

进入地点后，如果该地点有舰娘：

- 自动选中“互动优先级最高”的舰娘

后续交互指令默认作用于当前选中舰娘。

主界面状态区只需要重点展示：

- 玩家
- 当前选中舰娘

其余在场舰娘通过列表浏览，不需要同时展开完整状态。

## 地图层级结构

### Layer 1: 大区（Area）

最高层世界观分区，用于：

- 大范围导航组织
- 阵营归属
- 功能分区
- 天气与节日后续扩展

### Layer 2: 子区（SubArea）

大区内的中层聚类，用于：

- 组织同类地点
- 提供分流目标
- 容纳后续地图整改

### Layer 3: 具体地点（Location）

玩家真正可到达、可见舰娘、可触发事件和可执行指令的地点层。

当前玩法逻辑应主要以这一层为准。

### Layer 4: 地点槽位（Slot）

地点内部的系统分流信息，不是玩家额外导航层。

它不是隐藏地点，也不是玩家到不了的区域，而是同一地点内部的组织标签。

例如：

- 训练场 / 器械区
- 训练场 / 对抗区
- 宿舍大厅 / 东侧
- 码头 / 装卸区

第一版可以先保留这个概念，但不强制把它完全暴露给玩家 UI。

## 角色地图分布规则

角色位置由四层规则共同决定，而不是单一日程表。

### 1. 强制状态层

优先级最高：

- 委托中 -> 从地图移除
- 同行中 -> 跟随玩家
- 约会中 -> 跟随玩家或停留在约会地点
- 特殊剧情中 -> 进入专用地点
- 后续调教中 -> 进入私密场景

### 2. 工作安排层

如果命中当前时间条件下的工作安排：

- 优先进入工作地点
- 工作地点内部再按槽位或备选地点分流

### 3. 归属回流层

无工作时按稳定归属回流：

- 夜间优先回阵营生活区
- 深夜优先回宿舍群
- 宿舍内再按宿舍大厅、走廊、休息区等分布

### 4. 空闲分布层

在没有强制状态、没有工作安排时，角色会根据规则做轻量自由移动。

影响因素包括：

- 阵营归属
- 角色个性
- 当前时段
- 饭点需求
- 与玩家关系
- 玩家当前位置
- 当前地点承载情况

## 自动选中机制

当玩家进入一个地点且该地点有舰娘时，系统自动选中“互动优先级最高”的舰娘。

推荐优先级：

1. 功能型角色优先
   - 商店服务角色
   - 特殊系统入口角色

2. 有即时内容价值的角色优先
   - 当前地点/时间下有事件或差分口上的角色

3. 与玩家关系更高的角色优先

4. 最后按稳定顺序兜底
   - 例如 `actor_key`

## 数据结构建议

### AreaDefinition

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 大区 key |
| `display_name` | str | 大区名称 |
| `kind` | str | `public` / `faction_living` |
| `faction_key` | str? | 阵营生活区时对应阵营 |

### SubAreaDefinition

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 子区 key |
| `area_key` | str | 所属大区 |
| `display_name` | str | 名称 |
| `tags` | [str] | 功能标签 |

### LocationDefinition

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 地点 key |
| `sub_area_key` | str | 所属子区 |
| `display_name` | str | 名称 |
| `tags` | [str] | 功能标签 |
| `capacity_soft` | int | 建议承载人数 |
| `capacity_hard` | int | 最大承载人数 |
| `overflow_targets` | [str] | 超载分流目标 |
| `visibility` | str | `public` / `private` / `hidden` |

### LocationSlotDefinition

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 槽位 key |
| `location_key` | str | 所属地点 |
| `display_name` | str | 内部分组名 |
| `capacity_soft` | int | 软上限 |

### CharacterResidenceDefinition

建议角色定义后续补充：

| 字段 | 类型 | 说明 |
|------|------|------|
| `faction_key` | str | 阵营归属 |
| `residence_area_key` | str | 生活区归属 |
| `dorm_group_key` | str | 宿舍群归属 |
| `home_location_key` | str | 默认回流地点 |
| `default_activity_tags` | [str] | 空闲时偏好标签 |

## 与现有系统的关系

- 依赖 [地图与导航系统](地图与导航系统.md)
  - 当前平面地点系统后续需要升级为分层地图
- 依赖 [日程系统](日程系统.md)
  - 工作安排和空闲分布都要基于新地图层级
- 依赖 [时间与日历系统](时间与日历系统.md)
  - 饭点、夜间回流、节日活动区聚集都依赖时间语义
- 依赖 [委托系统](委托系统.md)
  - 委托状态直接影响角色是否出现在地图上
- 依赖 [商店与经济系统](商店与经济系统.md)
  - 商店将作为共享地点，后续接入明石/不知火服务语义

## 当前阶段不做

- 不做复杂 AI 行为树
- 不做逐分钟全角色重算
- 不做真实路径规划
- 不做所有槽位的玩家可见化
- 不立即实现明石/不知火商店入口逻辑

## 可扩展性要求

地图系统必须允许：

- 新增大区、子区、地点
- 调整旧地点归属
- 修改某个区域的层级而不推翻整个地图
- 增加新的承载规则或分流规则
- 在不重写内容系统的前提下接入新商店、新活动区或新剧情场景

## 记录策略

每次地图系统扩展后，应在两处同步记录：

### 1. `docs/TODO.md`

记录：

- 当前已完成的地图骨架范围
- 本轮新增的区域/地点
- 下一轮准备补哪块地图或分布逻辑

### 2. 本规格文档

更新：

- 当前已启用的大区/子区/地点
- 当前分流和自动选中规则
- 当前哪些区域已经接入真实玩法

## 实现顺序建议

1. 扩展地图数据结构到大区/子区/地点/槽位
2. 扩展角色归属字段
3. 实现地图分布调度器
4. 让委托状态从地图分布中移除角色
5. 接入地点内舰娘分页和自动选中机制
6. 最后再逐步接商店 NPC、天气、调教和活动场景
