# 指令系统

## 概述

指令系统管理玩家可执行的动作。每个指令定义了产出（SOURCE）、前置条件（地点/时段/关系/标记）和副作用（操作/施加标记）。指令执行必须经过 SOURCE → 结算管道，不得直接修改属性。

## 数据结构

### CommandDefinition

| 字段 | 类型 | 说明 |
|------|------|------|
| `key` | str | 唯一标识 |
| `display_name` | str | 显示名称（中文） |
| `category` | str | 分类：daily/work/intimacy/follow/date；空值表示未启用 |
| `location_tags` | [str] | 可用地点标签（OR 匹配，匹配任一即可） |
| `time_slots` | [str] | 可用时段 |
| `min_affection` | int? | 最低好感要求 |
| `min_trust` | int? | 最低信赖要求 |
| `min_obedience` | int? | 最低服从要求 |
| `required_stage` | str? | 最低关系阶段 |
| `requires_following` | bool? | 是否需要同行状态 |
| `requires_date` | bool? | 是否需要约会状态 |
| `required_marks` | {str: int} | 需要的标记（键=标记名，值=最低等级） |
| `remove_marks` | [str] | 执行后移除的标记 |
| `source` | {str: int} | SOURCE 产出 |
| `downbase` | {str: int} | 直接扣减的基础资源（例如 stamina/spirit） |
| `apply_marks` | {str: int} | 施加的标记（键=标记名，值=增量） |
| `operation` | str? | 副作用操作：start_follow/stop_follow/start_date/end_date |
| `success_tiers` | [float] | 成功分层倍率，供未来扩展掉落/文本分支使用 |

## 流程

### 指令可用性检查

```
1. 角色是否在当前地点？
2. 指令分类是否启用？（category 非空）
3. 全局模式检查：
   - 忙碌状态 → 禁止 daily/work/follow/date/intimacy
   - 约会途中 → 禁止 date 分类
4. 具体条件检查：
   - location_tags OR 匹配当前地点标签
   - time_slots 包含当前时段
   - min_affection/trust/obedience 满足
   - required_stage 对应的 rank 满足
   - requires_following/requires_date 满足
   - required_marks 全部满足
   - vitality 门槛满足（体力归零直接失败，气力不足则拦截消耗型动作）
```

检查顺序重要：先检查地点/时段（快速失败），再检查关系条件（需要查表），最后检查标记。

### 指令执行

```
1. 可用性检查 → 不通过则抛出 ValueError（含中文原因）
2. SceneService.build_for_actor() 构建场景上下文
3. command.source 写入 actor.stats.source
4. command.downbase 先交给 VitalService 处理，必要时扣减体力/气力
5. SettlementService.settle_actor() 执行结算
6. 构建 trigger_scene（结算后、操作前的快照，用于事件触发）
7. _apply_operation() 执行副作用（start_follow/stop_follow/start_date/end_date）
8. _apply_marks() 施加/移除标记（尊重 max_level 上限）
9. 构建 settled_scene（操作和标记后的快照）
10. EventService.triggered_events() 匹配事件
11. DialogueService.lines_for() 选择口上
12. 返回 ActionResult
```

**事件触发用的场景快照**：优先使用 trigger_scene（操作前），若无匹配则使用 settled_scene（操作后）。这确保状态转换事件（如 start_date）能正确触发。

## 配置

文件路径：`data/base/commands.toml`

指令分类体系：

| 分类 | 说明 |
|------|------|
| daily | 日常互动（闲聊、泡茶、训斥等） |
| work | 工作类（文书、训练、学习等） |
| intimacy | 亲密类（调戏、接吻、告白等） |
| follow | 同行类（邀请同行、告别、散步等） |
| date | 约会类（牵手、散步、送礼等，需约会状态） |

## 扩展方法

1. **加新指令**：在 `commands.toml` 中加 `[[commands]]`，填入 key/display_name/category/location_tags/time_slots/source/downbase
2. **加前置条件**：使用 required_stage/min_affection/requires_following 等字段
3. **施加标记**：使用 apply_marks 字段，如 `apply_marks = { drunk = 1 }`
4. **加副作用操作**：在 CommandDefinition 加 operation 字段，在 CommandService._apply_operation() 中添加新分支
5. **加消耗/门槛**：优先走 `downbase` 和 `VitalGate`，不要把体力气力逻辑散到指令文本里
