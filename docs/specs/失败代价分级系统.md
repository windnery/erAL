# 失败代价分级系统（草案）

更新时间：2026-04-22
状态：待产品确认

## 1. 目标

把“可尝试后失败”从提示文本，升级为可计算、可回归的负面结果系统。

本规格仅处理这类失败：

1. 指令可见且可点击。
2. 执行时因关系/数值/状态不足而失败。
3. 失败后必须产生负面后果。

本规格不处理这类情况：

1. 地点/时段/场景不符导致的不可见指令。
2. 系统配置错误、缺道具等工程错误。

## 2. 参考抽取依据

本草案基于以下文件中的负面数值分布抽取：

1. docs/reference/tw-command-numeric-catalog.md
2. docs/reference/tw-scom-numeric-catalog.md
3. CSV/source.csv（反感、恐怖、苦痛、逸脱、鬱屈等轴）
4. CSV/Palam.csv（反感等累积轴）
5. CSV/Mark.csv（反発刻印）

抽取观察（简化）：

1. 轻越界动作常见反感值在 10-80 区间。
2. 中越界动作常见反感值在 100-400 区间，常伴随恐怖/逸脱。
3. 强制或公开羞耻动作常见反感值在 500+，并常伴随高恐怖、高苦痛、高鬱屈。

## 3. erAL 当前约束（必须先认清）

### 3.1 已有负面链路

1. SOURCE 的 disgust/fear/pain/humiliation/deviation/frustration 已可结算到 PALAM。
2. SOURCE 的 disgust/fear 会抬升 BASE:anger（见 settlement_rules.toml）。
3. denial 已进入关系增长负项（favor_calc/trust_calc 的 base_negative_keys）。

### 3.2 当前缺口

1. commands.py 的门禁失败当前主要是 raise ValueError，不产生数值后果。
2. favor_calc/trust_calc 当前 clamp_min=0，意味着关系变化只会增加，不会因失败直接下降。

结论：

如果不先补“失败后果注入点”和“关系负增长通道”，就无法实现 TW 风格的失败代价。

## 4. 分级定义（草案）

### L1 轻失败（试探越界）

触发语义：

1. 轻度亲密动作尝试失败。
2. 角色不满但未进入强烈对抗。

默认后果模板（建议）：

1. SOURCE: disgust +20
2. SOURCE: frustration +20
3. SOURCE: humiliation +10
4. SOURCE: fear +0~10（可选）
5. BASE: mood 下调一个小档（如有情绪系统映射）

期望体感：

1. 角色表现出不快。
2. 玩家被警告“继续越界会更严重”。

### L2 中失败（明显越界）

触发语义：

1. 中强度亲密或推倒前动作失败。
2. 角色明确拒绝，进入警戒。

默认后果模板（建议）：

1. SOURCE: disgust +80
2. SOURCE: fear +50
3. SOURCE: humiliation +40
4. SOURCE: frustration +60
5. SOURCE: deviation +40
6. PALAM: denial +20（建议直接写入或通过专门映射写入）

期望体感：

1. 角色情绪明显恶化。
2. 后续可执行范围收紧。

### L3 重失败（强制/公开羞耻失败）

触发语义：

1. 强制动作失败。
2. 公开羞耻类动作失败。
3. 角色进入强烈对抗状态。

默认后果模板（建议）：

1. SOURCE: disgust +200
2. SOURCE: fear +150
3. SOURCE: pain +80
4. SOURCE: humiliation +120
5. SOURCE: frustration +150
6. SOURCE: deviation +120
7. PALAM: denial +60
8. 标记或状态：可触发反发相关标记（若已实现）

期望体感：

1. 角色明显生气/反抗。
2. 后续一段时间内互动难度显著上升。

## 5. 指令归类建议（第一批）

### 5.1 L1 候选

1. 常驻轻接触类（如轻度身体接触、逗弄类）。
2. 关系刚解锁边缘动作的早期尝试。

### 5.2 L2 候选

1. 身体接触中偏强动作。
2. 推倒前置动作。

### 5.3 L3 候选

1. 公开羞耻动作（如摄影、野外类）。
2. 强制类动作。

注：

具体指令清单应在 commands.toml 上按类别批量标注 failure_tier，不建议继续硬编码在 commands.py。

## 6. 数据设计建议

建议在 commands.toml 增加可选字段：

1. failure_tier = "l1" | "l2" | "l3"
2. failure_profile = "soft_reject" | "hard_reject" | "public_break"（可选）
3. failure_text_key = "..."（可选，用于文案表）

建议新增配置文件：

1. data/base/failure_penalties.toml

示例结构：

1. [tier.l1] / [tier.l2] / [tier.l3]
2. source_add = { disgust = 20, frustration = 20, ... }
3. palam_add = { denial = 20 }
4. mood_shift = -1
5. apply_mark = "..."（可选）

## 7. 实现顺序建议（必须按顺序）

1. 先做数据层：failure_penalties.toml + commands.toml 的 failure_tier 标注。
2. 再做执行层：CommandService 在“可尝试但失败”分支调用 penalty apply。
3. 再做关系层：允许失败后出现关系负增长（至少支持小幅下降）。
4. 最后做反馈层：统一失败文案风格（你已确认偏日常感）。

## 8. 验收标准（草案）

1. 同一条可尝试动作连续失败三次，玩家能感知到后果递增。
2. 失败后至少影响两类状态（例如反感 + 情绪，或反感 + denial）。
3. 失败后续影响可在下一次互动中体现（不是仅当回合文本）。
4. 有自动化测试覆盖 L1/L2/L3 各至少 1 个动作。

## 9. 待你拍板的点

1. L1/L2/L3 的默认数值档位是否接受。
2. 是否允许失败直接扣好感/信赖（目前系统默认只增不减，需要明确放开）。
3. 第一批纳入分级的指令名单（建议先 10-20 条高频动作）。
4. 是否把 denial 作为每个等级的固定后果之一。
