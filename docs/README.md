# docs 索引与权威边界

## 目的

本文件用于防止文档混用，明确：

1. 哪些文档是执行依据。
2. 哪些文档仅供参考。
3. AI 和开发者应按什么顺序阅读。

## 阅读顺序（AI 与人共用）

1. 先读 `docs/README.md`（本文件）。
2. 再读 `docs/AI_LONG_TERM_VIBE_GUIDE.md`（执行规则与质量门）。
3. 再读 `docs/PRD.md`（目标与验收）。
4. 再读 `docs/TODO.md`（任务状态与排期）。
5. 如涉及指令迁移，再读 `docs/COMMAND_MIGRATION.md`。
6. 如涉及架构边界，再读 `docs/architecture-plan.md`。
7. `docs/reference/` 与 `docs/superpowers/` 仅在需要细节时查阅。

## 文档分级

### A 级：权威执行文档（可直接驱动开发）

| 文档 | 角色 | 是否执行依据 | 更新触发 |
| --- | --- | --- | --- |
| `docs/AI_LONG_TERM_VIBE_GUIDE.md` | 开发治理规则、质量门、AI 协作协议 | 是 | 执行规范变化、质量门变化 |
| `docs/PRD.md` | 产品目标、范围边界、验收标准 | 是 | 目标范围或验收变化 |
| `docs/TODO.md` | 任务看板、优先级、状态 | 是 | 每次任务状态变化 |
| `docs/COMMAND_MIGRATION.md` | 指令迁移计划与进度 | 是（仅指令域） | 新增/调整指令迁移 |

### B 级：架构说明文档（受控执行依据）

| 文档 | 角色 | 是否执行依据 | 更新触发 |
| --- | --- | --- | --- |
| `docs/architecture-plan.md` | 架构演进计划与设计说明 | 条件性（需与 A 级一致） | 架构调整或重构计划变化 |

说明：当 B 级与 A 级冲突时，以 A 级为准，先修正文档再动代码。

### C 级：参考资料（不可直接作为任务指令）

| 目录/文档 | 角色 | 是否执行依据 |
| --- | --- | --- |
| `docs/reference/` | eraTW/迁移矩阵/索引资料 | 否 |
| `docs/superpowers/` | 计划草案与辅助资料 | 否 |

## 冲突处理规则

1. 若 `TODO` 与 `PRD` 冲突：以 `PRD` 验收口径为准。
2. 若任务与 `AI_LONG_TERM_VIBE_GUIDE` 冲突：以质量门和治理规则为准。
3. 若 `architecture-plan` 与执行文档冲突：先改文档口径，再推进实现。
4. 任何冲突都必须在提交说明中记录“冲突 -> 决议 -> 影响范围”。

## 最小维护要求

1. 每周至少一次检查 A 级文档是否口径一致。
2. 每次大改后更新本文件中的文档分级或阅读顺序。
3. 新增 docs 文档时，必须在本文件登记其级别和用途。

## 新增文档登记模板

```markdown
- 文档：docs/xxx.md
- 级别：A / B / C
- 用途：
- 是否可直接驱动开发：是 / 否
- 维护频率：按需 / 每周 / 每次迭代
```
