# erAL MVP PRD

## 1. 项目定位

`erAL` 是一个以碧蓝航线港区为题材、优先直接迁移 `eraTW` 通用玩法系统、并借鉴 `erArk` 工程组织的单机原型项目。

本阶段目标不是复制 `eraTW` 的全部题材内容，而是优先把它已经成熟的通用玩法层直接迁到港区题材里，先做出一个内容量较小、但玩法感觉接近 `eraTW` 的可玩版本：玩家能在港区里按时段找人、移动、互动、推进关系、触发事件、进入约会与私密场景，并且这个循环可以持续重复游玩。

## 2. MVP 定义

### 2.1 玩家体验目标

1. 玩家以天数和时段为节奏在港区内行动。
2. 角色会按日程出现在不同地点，玩家需要主动“找人”和“追人”。
3. 指令受地点、时段、关系阶段、同行状态、约会状态、MARK 烙印共同限制。
4. 指令先产出 `SOURCE`，再统一结算到 `BASE / PALAM / CFLAG / TFLAG` 与显式状态字段。
5. 每次行动都可能带来事件命中、文本变化、关系推进和新玩法解锁。
6. 玩家可以从日常互动推进到同行、约会、轻亲密互动，但 MVP 不进入复杂 H 流程。

### 2.2 MVP 完成标准

- 1 个港区小地图，至少 8 到 12 个地点，覆盖公共、工作、餐饮、训练、私密等场景。
- 3 到 5 名可互动角色，每名角色都有独立 `character.toml`、日程、事件和对话。
- 20 到 30 个可用指令，覆盖日常、工作、同行、约会、轻亲密互动。
- 5 段关系阶段：`陌生 -> 友好 -> 喜欢 -> 爱 -> 誓约`。
- MARK 系统参与前提判断、事件触发和文本分支。
- 前期所有角色共用通用口上层，后期只为少量核心角色补专属口上。
- CLI 支持移动、等待、查看状态、执行指令、存档、读档、退出。
- 至少支持连续游玩 7 天，并能稳定完成 1 条约会线与 1 条轻亲密线。
- 静态内容继续保持数据驱动，后续新增角色时不需要改核心系统代码。

### 2.3 MVP 非目标

- 不做全量港区大地图。
- 不做 800+ 角色接入。
- 不做图形前端，当前阶段以 CLI 为准。
- 不做复杂 H 流程、深层 AI 日程模拟或大规模剧情树。
- 不追求 `eraTW` 的内容体量，只优先迁移它成熟的通用玩法系统与推进手感。

## 3. 核心玩法

### 3.1 单日循环

1. 查看当前时段、地点和可见角色。
2. 根据地图可见性规则移动到目标地点。
3. 在当前地点对角色执行可用指令。
4. 指令通过 `SOURCE -> settlement` 推进角色状态。
5. 命中事件后优先播放事件文本，否则走动作兜底文本。
6. 选择继续互动、邀请同行、发起约会，或等待进入下一时段。

### 3.2 中期推进循环

1. 通过闲聊、共同行动、约会等方式提高好感、信赖、服从。
2. 关系阶段提升后解锁更多指令、地点和文本分支。
3. 在同行或约会状态下进入私密地点，体验更深一层的互动。
4. 通过 MARK 和记忆型事件让角色关系产生可追踪变化。

### 3.3 核心系统支柱

- 地图与移动：地点是玩法变量，不只是背景。
- 时段与日程：角色会移动，玩家必须围绕日程安排行动。
- 指令与结算：优先直接继承 `eraTW` 的指令语义、数值语义和 `SOURCE -> settlement` 管线。
- 数值轴：`BASE / TALENT / SOURCE / PALAM / ABL / FLAG / CFLAG / TFLAG` 这类通用内容优先直接迁移。
- 关系与 MARK：关系阶段采用碧蓝航线风格命名，MARK 决定细分分支。
- 事件与文本：系统判定先于文案，文案按事件和条件分层选择。
- 角色包：角色内容应持续通过数据包扩展，而不是把剧情硬写进系统层。

## 4. 技术栈

- 语言：`Python 3.11+`
- 依赖：标准库优先，当前不引入第三方运行时依赖
- 包结构：`src/` layout + `setuptools`
- 运行入口：`python -m eral`
- 测试：`unittest`
- 静态数据：`TOML` + `JSON`
- 工具链：`import_tw_axes`、`validate_content`
- 表现层：CLI
- 运行时目录：`runtime/saves`、`runtime/logs`、`runtime/cache`

## 5. 编码约定

### 5.1 语言与命名

- 用户可见文本、注释、文档统一使用中文。
- 代码标识符、数据 key、文件名使用英文 `snake_case`。
- `display_name` 使用中文，`key` 保持稳定英文标识。

### 5.2 分层边界

- `app` 只做装配和配置，不写玩法逻辑。
- `engine` 只放题材无关基础设施。
- `domain` 只定义世界状态和模型，不写结算逻辑。
- `systems` 负责编排规则和修改状态。
- `content` 只做静态数据加载，不写运行时规则。
- `ui` 只能消费系统结果，不能直接篡改核心状态。

### 5.3 数据驱动规则

- 指令、地图、关系阶段、结算规则、事件、对话、MARK 都优先外置到数据文件。
- 角色内容按角色包组织，每个角色独立维护自己的 `character.toml`、`events.toml`、`dialogue.toml`。
- 不把剧情文案和规则判断硬编码在同一个系统文件里。
- `eraTW` 中已经稳定的通用系统优先直接迁移语义、命名和数值设计，避免重复设计。
- 不迁移 `eraTW` 的裸数组直写、脚本式隐式耦合和低可维护实现习惯。

### 5.4 状态修改规则

- 指令不得直接跨过 `SOURCE` 修改主要数值。
- 好感、信赖、服从等高频状态允许通过显式字段访问，但权威来源仍需可追溯到兼容层或统一结算结果。
- 同行、约会、遭遇、MARK 等状态必须有明确的读写入口，不能依赖隐式约定。

### 5.5 测试与交付

- 新增系统时必须补对应测试；新增指令、事件和关键分支时至少补一条集成测试。
- 新增角色包或修改内容格式时必须跑内容校验。
- 文档与实现一起维护；涉及玩法借鉴范围变化时同步更新 `docs/reference/` 下相关说明。
