# erAL 产品需求文档（Product Baseline V1）

更新时间：2026-04-22

本文档取代旧 MVP 口径，作为当前阶段唯一产品目标文档。

## 1. 产品定位

erAL 是一个以碧蓝航线港区为题材、继承 eraTW 玩法语义并采用 Python 数据驱动架构实现的单机角色互动游戏。

当前阶段目标不是“先做一个最小可玩”，而是“让已实现系统完整表达并可被玩家感知”。

## 2. V1 目标（本阶段）

### 2.1 目标定义

V1 的目标是完整表达，不是趣味优化。

完整表达指：

1. 已实现系统都能进入真实运行链路，而不是只停留在数据层或显示层。
2. 玩家能在一次正常游玩中感知到系统的输入、变化和反馈。
3. 系统间衔接清晰，不出现大量“可配置但无消费者”的空洞能力。

### 2.2 V1 完成标准

满足以下条件即可视为 V1 达成：

1. 主链完整：指令 -> 结算 -> 关系 -> 事件 -> 口上 -> 时间推进 -> 存档。
2. 表达完整：每个已落地子系统都能映射到至少一个玩家可见反馈。
3. 数据闭环：新增字段必须有消费者，新增系统必须有测试与文档。
4. 入口一致：运行方式、文档描述、实际表现层三者一致。
5. 规模真实：文档中的系统/内容规模与仓库实际统计一致。

## 3. 当前基线（2026-04-22）

1. 系统模块：35
2. 指令条目：162
3. 角色包：5

说明：以上数字用于统一口径，不直接代表质量达标。

## 4. 核心体验闭环

### 4.1 单回合体验

1. 玩家在当前地点与时段查看可执行行为。
2. 门禁决定行为是否可执行。
3. 执行后产出 SOURCE 并统一结算。
4. 命中事件时播放事件口上，未命中走动作兜底口上。
5. 时段推进并刷新角色位置、环境状态与后续可执行集。

### 4.2 多日体验

1. 玩家通过重复互动推进关系阶段。
2. 关系推进解锁更深层行为与文本。
3. 经济、设施、赠礼、皮肤、持续状态提供中长期目标。
4. 存档保证中断后可继续。

## 5. 范围边界

### 5.1 本阶段纳入范围

1. 系统表达完整化（把已做系统真正接入玩家体验）。
2. 内容与系统联动补齐（事件、口上、门禁、数值消费者）。
3. 文档系统重建与维护机制落地。
4. 回归测试与内容校验增强。

### 5.2 本阶段非目标

1. 不以“好不好玩”作为第一优先级。
2. 不追求一次性扩充到超大角色体量。
3. 不强行开启全新前端重构作为前置条件。
4. 不为了新增系统而新增系统，优先补现有系统表达闭环。

## 6. 架构与实现约束

1. 分层不变：app/engine/domain/systems/content/ui。
2. 指令不得绕过 SOURCE 直接写核心数值。
3. 规则优先数据驱动，不把条件与文案硬耦合在代码里。
4. UI 只消费系统结果，不直接篡改核心状态。
5. 新增内容必须通过 validate_content 与基础回归测试。

## 7. 验收指标（V1）

1. 功能验收：主链可连续运行，核心路径无阻断。
2. 表达验收：每个已实现系统在 `docs/玩家体验映射.md` 中有明确玩家感知条目。
3. 一致性验收：README、PRD、TODO、系统清单无互相矛盾口径。
4. 质量验收：核心流程有稳定自动化回归，新增改动不会频繁破坏旧链路。

## 8. 文档协同规则

1. 范围变化先改 PRD，再改 TODO 与系统清单。
2. 系统行为变化先改代码，再改 specs 与体验映射。
3. 每次里程碑结束同步更新基线数字。
