# erAL MVP 开发表

更新时间：2026-04-26

目标：基于当前已经完成的结算管线、数据契约、角色包、基础运行链路，做出一个“能连续玩、能感知反馈、能回收验证”的 MVP。这个 MVP 不是最少功能集合，而是最小完整闭环。

## 1. 当前已完成底座

以下内容视为已完成资产，MVP 期间直接复用，不作为开发项重复回头做：

1. 应用启动与装配。
2. 角色包加载与基础数值初始化。
3. 体力/恢复系统基础行为。
4. 存档与读档基础流程。
5. 指令数据契约：`train.toml`、`command_effects.toml` 的 SOURCE 主链。
6. 结算管线：`SOURCE -> SOURCE_EXTRA -> SOURCE_CBVA -> CUP -> PALAM/CFLAG -> JUEL -> 刻印 -> ABLUP`。
7. 结算入口：`SettlementService.settle_actor()` 单入口。
8. 当前测试基线与回归入口。

## 2. MVP 定义

MVP 的完成标准不是“内容数量够不够”，而是下面这条主链能否连续成立：

1. 指令可用性可以被判定。
2. 指令执行后能产出 SOURCE 和必要副作用。
3. 结算链路能把结果写进关系、刻印、成长和反馈。
4. 事件与口上能读取场景上下文并给出可见反馈。
5. 时间推进后，角色状态与可执行集会更新。
6. 存档/读档后状态保持一致。

## 3. MVP 范围

### 3.1 本阶段必须完成

1. 一个完整可玩的主循环。
2. 主循环所需的最小命令集。
3. 关系、事件、口上、时间推进、存档的闭环。
4. 主链上已有产出字段的消费者补齐。
5. 至少一轮完整回归测试，证明闭环不被破坏。

### 3.2 本阶段后置

1. 大规模扩角色。
2. 前端重构。
3. 全量 TW 细碎文件一比一复刻。
4. 结算主链之外的高复杂度支线系统。

## 4. 开发节奏

### Phase 0：收口确认

1. 冻结主链边界和验收标准。
2. 确认 MVP 只围绕一个玩家可连续执行的回合展开。
3. 把现有完成项从任务表里移到“已完成资产”。

### Phase 1：主链闭合

1. 补齐指令可用性与拒绝反馈。
2. 补齐命令效果里的 vitals / experience。
3. 补齐场景上下文到事件和口上的读取。
4. 补齐关系、刻印、ABL 的实际消费者。

### Phase 2：可玩回合

1. 选择一个现有内容密度最高的回合路径。
2. 让这个回合能从指令开始，走到反馈、时间推进、存档恢复。
3. 让玩家能重复执行同一循环而不崩链。

### Phase 3：内容密度

1. 追加最少量事件和口上变体。
2. 补齐最常用角色的基础反馈。
3. 让关键数值变化在玩家视角可感知。

### Phase 4：稳定化

1. 跑通回归测试。
2. 修掉闭环内的阻断问题。
3. 把已完成内容同步回写到 `TODO.md`、`系统清单.md` 和 `玩家体验映射.md`。

## 5. 任务表

### A. 必做主链任务

| 优先级 | 任务 | 目标 | 依赖 | 验收标准 |
|---|---|---|---|---|
| P0 | 指令可用性判定收口 | 让命令进入“能用/不能用/为什么不能用”三态 | 现有 command gates | 至少覆盖体力、关系、刻印、装备四类拒绝理由 |
| P0 | vitals 填充 | 让命令执行有体力/气力消耗 | `command_effects.toml` | 主循环命令都有合理体力消耗值 |
| P0 | experience 填充 | 让命令执行能驱动成长 | `command_effects.toml` | 主循环命令都有合理经验值 |
| P0 | 场景上下文反馈 | 让事件/口上读到当前回合信息 | `SceneContext` | 至少一个事件和一组口上能命中场景状态 |
| P0 | 关系消费者补齐 | 让关系增长真的影响行为 | `relationship_growth.toml` | 关系阶段能影响可用指令或文本 |
| P0 | ABL 消费闭环 | 让 ABL 升级后能改变玩法 | `abl_upgrade.toml` | 至少一个指令或门禁读取 ABL |
| P0 | 刻印可见反馈 | 让刻印等级进入玩家视角 | `marks.toml` / 口上 | 至少一处玩家反馈能显示刻印变化 |
| P0 | 存档后状态一致 | 让循环可中断恢复 | 存档系统 | 读档后主链状态不丢失 |

### B. MVP 内容任务

| 优先级 | 任务 | 目标 | 依赖 | 验收标准 |
|---|---|---|---|---|
| P1 | 选择一个主可玩场景 | 把系统收束到单次可玩回合 | 现有角色包/地点/时间 | 玩家能重复跑同一场景闭环 |
| P1 | 基础事件补齐 | 让主链里有事件反馈 | `events.toml` | 回合中能命中至少一类事件 |
| P1 | 口上变体补齐 | 让反馈不再全是默认句 | `dialogue.toml` | 至少覆盖“开始/拒绝/成功/结束”四类反馈 |
| P1 | 主角/常用角色补全 | 让 MVP 内容看起来像完整产品 | 现有角色包 | 核心角色至少有基本事件和台词 |

### C. MVP 之后再做

| 优先级 | 任务 | 说明 |
|---|---|---|
| P2 | 体位倍率扩展 | 属于结算精细化，不影响 MVP 主闭环 |
| P2 | 高潮/反击/体液系统 | 属于支撑系统扩张，放到闭环后 |
| P2 | 大规模角色扩充 | 先让一个闭环成立，再扩内容宽度 |
| P2 | UI 大重构 | 不是当前 MVP 的前置条件 |
| P2 | TW 全量细碎文件复刻 | 只作为长期完成度目标，不作为当前任务颗粒度 |

## 6. 执行规则

1. 任何新发现的依赖，只追一跳，不继续向上游无限展开。
2. 任何新增字段必须有消费者，不能只停在数据层。
3. 任何完成一轮闭环后，先回写文档，再开下一轮。
4. 如果某任务不能提升当前闭环完成度，就放到后续队列，不进入本轮开发。

## 7. 建议执行顺序

1. 先完成 P0 主链任务。
2. 再做 P1 内容任务。
3. 最后把剩余内容按 P2 队列推进。

## 8. 备注

1. 这个 MVP 的目标是把你已经完成的底座真正接到玩家体验上。
2. 长期目标仍然是向 TW 那种完成度推进，但推进方式改成“先闭环，再扩展”。

## 9. 现有 TODO 的处理方式

1. `TODO.md` 继续承接系统/代码侧任务，但只保留会直接推动 MVP 主链闭合的内容。
2. `TODO_ME.md` 继续承接数据/内容侧任务，但只保留会直接喂给 MVP 主链的字段和内容。
3. 已完成底座不要再写进两份 TODO 里，改为在这里引用和复用。
4. 新发现的依赖如果不会影响当前主链闭环，统一下沉到 P2 之后，不在本轮展开。
